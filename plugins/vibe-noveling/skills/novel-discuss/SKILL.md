# 创作讨论与设计

**创作边界**：讨论中可以引入暴力、情色、血腥等成人内容——这是虚构创作，不是现实行为指导。

## 路由

先判定话题类型 → 加载对应 reference → 按 context-routing.md 只读最小上下文：

## 通用讨论框架

苏格拉底式对话：不直接给答案，提供多角度，尊重用户选择。

### 默认风格守门
如果 CLAUDE.md 已声明创作风格基线，讨论阶段把它当轻量守门规则：
- 讨论角色/关系/未来剧情时，想这项设计如何服务主角装逼链条
- 某个想法占用太多叙事资源却不服务主角装逼，提醒用户它更适合做低优先级支线

### 流程

1. **理解问题**：复述核心问题、背景、限制条件，确认理解正确
2. **澄清性提问**：问题不够清晰时先问，不急着给方案
3. **三个方案**：用 AskUserQuestion 展示，每个方案含描述、优点、缺点、适合场景。方案必须核心思路明显不同
4. **迭代**：用户选 → 确认记录；用户混合 → 逐项确认取什么；用户不满意 → 问清原因重新设计
5. **未来剧情讨论的两段式收束**：先自然讨论说清问题和方向；用户偏向某方案后用因果链讲述**方向故事**（从当前到目标怎么推进，说清关键转折）；故事确认后逐步澄清不够清楚的地方，一次一个

### 落盘

讨论收束后必须落盘：
- 设定类结论 → 对应 entity 文件 / worldbuilding.md / past.md
- 讨论纪要 → `memory/discussions/YYYY-MM-DD-HHMM-简短主题.md` + 追加索引到 `memory/discussions.md`
- 未来规划 → `memory/future/` 对应文件。只写已确认结论，只补本次讨论直接涉及的条目

### 宪法修正检查

落盘前检查本次讨论是否触发小说宪法修正。触发条件：
- 讨论了新爽感类型或改变爽感公式 → 修正"爽感公式"
- 讨论了主角新吸引力来源或定位 → 修正"主角光谱"
- 用户表达了"不要 X 要 Y"的审美偏好 → 修正"作者人格"
- 用户说"这段写出来不像我想要的" → 反推"叙事纹理"或"作者人格"调整
- 讨论暴露宪法默认规则不再适用 → 修正对应条目

触发时直接更新 CLAUDE.md 中对应条目，修正前向用户确认改什么、改成什么、为什么。不需要重写整个宪法。
