# 剧情规划

## 目标

通过两种模式帮助用户完成单章工作：

- **新规划模式**：先定节奏职责、再定任务、最后写出可手改的精简简纲
- **修订模式**：先读取已有大纲、再界定变更边界、最后做差分修改
- **正文测试**：用户确认大纲后，默认调用一个 Opus agent 真写一段正文，再根据试写结果反推大纲问题

本技能只负责把这一章规划清楚，不负责把这一章切成写作单元。写作单元切分发生在 `novel-write` 阶段内部。

## 使用方式

```bash
/novel-plan
/novel-plan --revise
```

## 新契约

### 你现在要产出的是什么

`ch-xxxx-outline.md` 必须是一份适合作者直接阅读和手改的章节简纲，而不是给写作 agent 看的执行表。

**必须包含**：

- 本章节奏定位
- 章节任务卡
- 起始状态
- 结束目标
- 第三人称精简剧情纲要
- 伏笔处理

**不再输出**：

- 剧情点编号
- 情绪走向栏位
- 场景类型栏位
- 字数配额
- 面向写作 agent 的细碎执行抓手

### 你现在不该做什么

- 不要把本章切成 10-20 个剧情点
- 不要把正文阶段才需要的切分结果写回 `outline.md`
- 不要为了“方便写作”牺牲“方便人工修改”

## 执行流程

### 前置：初始化检查

1. 检查 `memory/future/00-index.md` 和 `memory/future/10-book.md` 是否存在。不存在则建议先 `/novel-bookplan`。
2. **确定章节编号**：从 `CLAUDE.md` 的「创作进度 → 当前章节」读取，或由用户指定（如 `/novel-plan 5`）。后续流程中章节文件一律使用四位补零格式：`ch-xxxx-outline.md` / `ch-xxxx-context.md`。
3. **确定信息源优先级**：
   - 新规划模式：
     - `memory/future/00-index.md` + `memory/future/10-book.md` + `memory/future/20-threads.md` + 相关 `30-volumes/` / `40-arcs/`
     - `chapters/ch-{prev_padded}-outline.md`
     - `memory/past.md`
     - `chapters/ch-{prev_padded}-context.md` 只供 `novel-write` 使用，不反向主导规划
   - 修订模式：
     - `chapters/ch-{current_padded}-outline.md`
     - 用户明确提出的修改目标
     - `memory/future/10-book.md` + `memory/future/20-threads.md` + 相关 `30-volumes/` / `40-arcs/`
     - `chapters/ch-{prev_padded}-outline.md`
     - `memory/past.md`
     - `chapters/ch-{current_padded}-context.md` 只用于判断缓存是否失效

如果高优先级文件缺失，明确提示缺哪一个；不要擅自用低优先级信息替代。

### 前置：模式判定

- 用户要从零设计下一章，或当前章还没有 outline → **新规划模式**
- 用户明确说要在已有大纲上调整、修改、补桥接、改结尾、重写后半章 → **修订模式**
- 当前章已有 outline，且用户指出“这里太突兀 / 后半章不顺 / 收尾想改” → **修订模式**

如果进入修订模式，默认不回到从零发散；只有现有骨架失效，或用户明确要求大改时，才退回新规划模式。

## 第一段：定节奏

### 第一步：读取上下文

输出当前进度、已完成章节、待处理伏笔、当前卷信息，并整理起始状态：

```text
📖 规划上下文

当前进度：
   卷：Vol-01（初入修真）
   已完成章节：第 10 章
   当前境界：炼气三层
   当前位置：凌霄宗外门

最近完成：
   第10章：演武场击败王师兄

待处理伏笔：
   - 空灵根的秘密（Vol-03）
   - 神秘势力的关注（Ch 20）

【起始状态】
   • 境界：炼气三层 | 位置：凌霄宗外门
   • 资源：灵石 50
   • 关系：敌对-王师兄，友好-李师姐
   • 近期：演武场切磋，展现不俗实力
```

**衔接检查**：

读取上一章大纲的结束目标、章末钩子和最后一个关键动作，与本章起始状态对比。发现不一致时明确标注缺哪一步过渡。

**章节伏笔衔接检查**：

读取上一章大纲中的伏笔计划，筛选出状态为「⏳ 待回收」或「🔄 已移交追踪表」的伏笔，并提示本章是否处理。

如果卷级节奏信息不足，明确提示：

```text
⚠️ 卷级节奏信息不足

我可以继续规划单章，但这会降低和整卷结构的贴合度。
更稳妥的做法：先回 /novel-bookplan 补该卷的节奏卡，再回来继续。
```

### 第二步：生成本章节奏定位

在章节任务卡之前，先输出本章在全书/分卷中的结构位置。

```text
📍 本章节奏定位

所属卷：Vol-02
主母型：Institutionalized
副母型：Fool Triumphant
卷内位置：中前段
承接的全书 beat：Fun and Games 区间

本章主职责：
1. 兑现这一卷的题材承诺
2. 推进主角在组织内部的站位变化
3. 为卷中点反转埋预热

本章不负责：
1. 提前揭开终极真相
2. 提前打完本卷 Boss
3. 一次性回收所有上一章伏笔
```

如果本章只是承上启下，也要直说，不要硬编成大转折章：

```text
本章类型：过桥章
结构价值：把角色从 A 状态送到 B 状态，同时保留钩子
```

## 第二段：定任务

### 第一步：生成章节任务卡

基于 future/ 的章节定位、分卷节奏卡、当前 arc、上一章结束状态、待回收伏笔，先提出一个可执行的章节任务卡，再让用户确认。

```text
📌 章节任务卡

本章主任务：进入五丰镇并完成第一次有效对接

必须推进的线：
1. 与周华顺建立联系
2. 展示夜市与鲁班坊体系
3. 让叶芷恒完成第一次有效出场

允许延期的线：
1. 空灵根暗线继续压后
2. 通讯阵更深层异常先不揭开

建议收尾点：苏寒染说“我要验验”
```

如果用户要改，优先改任务卡，不要直接跳到展开层。

### 第二步：确认本章范围和收尾点

在任务卡确认后，再确认本章收尾点和大致篇幅：

```text
📐 章节范围建议

建议收尾点：苏寒染说“我要验验”
字数预估：2800-3600 字

这个收尾点可以吗？还是想提前/延后？
```

## 第三段：做展开

### 第一步：一句话走向

以下流程默认用于**新规划模式**。修订模式只有在现有骨架已失效，或用户明确要求重做后半章/整章时，才回到这一段。

先检查 `memory/future/40-arcs/` 或对应卷卡中，本章是否已有明确事件规划或章节职责段。

**有明确规划** → 先复述职责，再确认是否沿用。  
**没有明确规划** → 给出 10 个可能走向，让用户选 1-2 个方向，不使用 AskUserQuestion。

用户选择或确认后，只围绕这三个维度补问，且一次只问一个：

1. 本章涉及哪些角色
2. 本章结束时角色状态会怎样变化
3. 本章需要承接前章的哪些悬念/线索

### 第二步：场景化方案

先选择叙事模板，再基于选定走向和叙事模板设计 3-6 个场景化方案。场景化方案只用于帮助你想清楚结构，不直接写入大纲。

不要把工具箱应用说明、novel-master 决策记录或过程性旁注暴露给用户。

### 第三步：设定完整度与命名一致性检查

在写入正式大纲前，先确认本章涉及的关键设定是否齐全。

**执行方式**：

1. 从选定方案中提取涉及的角色、地点、物品/概念
2. 检查 `memory/entities/` 下的对应设定文件是否存在且包含必填字段
3. 检查本章中计划使用的称呼是否能映射到已有实体

**检查范围**：

- 角色：基本信息、外貌描述、核心性格、语言风格、动机、修为境界、人物关系
- 地点：基本描述、环境/氛围、特点
- 物品/概念：基本描述、功能/效果、限制/条件

**输出完整度报告**：

```text
📖 设定完整度报告

【苏寒染】主角 | ✅ 完整
【凌绝尘】配角 | ⚠️ 语言风格缺失
【周华顺】配角 | ❌ 无设定文件
【五丰镇夜市】地点 | ⚠️ 仅有基本描述
```

**输出命名一致性报告**：

```text
🪪 命名一致性检查

【叶芷恒】canonical name 已存在
- 本章称呼：叶姑娘 | ✅ 可用（首次见面，视角合理）
- 本章称呼：叶芷恒 | ⚠️ 需确认是否已经知名
- 本章称呼：那少女 | ✅ 可用（短暂过渡）
```

如果发现设定缺失，立即停止，写入 `chapters/ch-{chapter_padded}-missing.md`，并提示用户去另一个 session 用 `/novel-discuss` 补全。

### 第四步：生成第三人称精简剧情纲要

这是本技能最关键的输出步骤。

你要写的是一份**第三人称精简剧情纲要**，不是执行表，也不是写作分镜。

#### 写法要求

- 主体使用连续短段落书写，建议 3-6 段
- 每段都写“正在发生什么”，而不是评论“这一段的作用”
- 必须使用第三人称转述
- 优先写动作、冲突、信息变化、关系变化
- 允许自然分段，但不做剧情点切分
- 方便作者手动增删、改写、搬动段落

#### 禁止事项

- 不要写“剧情点1 / 剧情点2”
- 不要写面向写作 agent 的过程性说明
- 不要堆砌“氛围、情绪走向、场景类型、字数”等执行字段
- 不要把一句话剧情硬拆成很多细点

#### 推荐结构

```text
## 基本信息
## 起始状态
## 结束目标
## 章节任务卡
## 第三人称精简剧情纲要
## 伏笔处理
```

#### 示例

```text
## 第三人称精简剧情纲要

苏寒染入城后先被夜市的秩序和热闹吸住视线，但他很快发现这里的繁华并不干净。周华顺带着生意人的圆滑靠近，把便宜、门路和熟人面子一起端到他面前，像是在帮忙，实则一点点试探他的底细。

苏寒染没有急着拆穿对方。他顺着周华顺的话往前走，一边看摊位和器坊，一边借细节判断这地方的规矩和漏洞。叶芷恒在这个过程中第一次出现。她没有直接站到任何一边，只用几句克制的话把场面轻轻拨了一下，让苏寒染意识到周华顺并不是这里唯一懂门道的人。

等周华顺把人带到更深的位置，试探开始露出真形。苏寒染先装作被牵着走，随后在验货和问价里反手把信息扣了回来。他确认自己确实被人当成了肥羊，也确认这件事背后不只是普通的坑骗。到章节结尾，他不再只是记住这笔账，而是明确决定亲手验一验这条线到底通向谁。
```

### 第五步：写入文件并等待确认

写入 `chapters/ch-{chapter_padded}-outline.md` 后提示用户阅读：

```text
📋 大纲已写入 chapters/ch-{chapter_padded}-outline.md

这份大纲现在是第三人称精简剧情纲要，适合你直接手动修改。
如果需要调整，我们先确认改动边界，再更新文件。
```

## 修订模式（基于已有大纲调整）

### 第一步：读取当前章并生成修订简报

先读取当前章节的已有大纲，提炼：

- 本章当前职责
- 起始状态和结束目标
- 现有剧情骨架
- 已安排的伏笔回收/埋设
- 当前版本最可能出问题的地方

输出格式：

```text
🩺 当前章修订简报

当前骨架：
1. 前半章先完成入场和信息对接
2. 中段发生被骗/试探
3. 结尾落在主角决定反击

建议保留：
1. 夜市初见的世界观展示
2. 周华顺的第一次有效接触

待解决问题：
1. 中段转折过快
2. 叶芷恒出场偏功能化
3. 结尾钩子不够利落
```

### 第二步：提出修订策略

不要让用户从零描述怎么改。先给 2-3 个修订策略，并推荐一个最稳的。

### 第三步：确认变更边界

确认时把边界说具体：

```text
📏 本次修订边界

保留：
1. 开场到夜市入场
2. 周华顺第一次接触

删除/弱化：
1. 原中段里的直接冲突

新增：
1. 一个被骗前的试探场景

收尾：
1. 从“先记下了”改成“我要验验”
```

### 第四步：生成修订清单并更新已有大纲

更新规则：

1. 优先保留未受影响的段落，不为了统一风格整章重写
2. 改的是局部，就只动局部
3. 如果结束目标、伏笔处理、涉及角色已被改动，要同步更新对应板块
4. 精简剧情纲要可以直接改段落顺序，但不要把它重新膨胀成剧情点表

### 第五步：输出联动检查

修订完成后，必须追加一轮联动检查：

```text
🔎 修订后联动检查

上一章承接：
   ✅ 起始状态仍能接上

本章职责：
   ✅ 仍然承担 Fun and Games 的兑现

伏笔状态：
   ⚠️ 原计划本章回收的[伏笔A]已延期到下一章

下一章承接：
   ⚠️ 如果下一章仍按旧计划开头，需要把第一场改成“验货/对质”
```

如果已有 `chapters/ch-{current_padded}-context.md`，且本次修订影响了以下任一项，视为缓存可能失效：

- 出场角色变化
- 关键地点变化
- 伏笔状态变化
- 本章结束目标变化

## 用户确认后执行的后续步骤

用户确认大纲后，按顺序执行以下两步（参考 `references/output.md`）：

### 第一步：尝试生成上下文缓存

调用 context-collector agent（`subagent_type: "context-collector"`），生成 `chapters/ch-{chapter_padded}-context.md`。

```text
为第 {chapter} 章生成上下文缓存。

大纲文件：chapters/ch-{chapter_padded}-outline.md

请根据大纲中的角色、地点、物品，收集完整的上下文信息。
将输出保存到 chapters/ch-{chapter_padded}-context.md。
```

如果 context 生成失败，明确提示：

```text
⚠️ 大纲已完成，但 chapters/ch-{chapter_padded}-context.md 尚未生成。
后续写正文前，请重新执行 /novel-plan 或单独补跑 context 收集。
```

### 第二步：执行 Opus 正文写作测试

在 context 生成完成后，或确认本次只能用 outline 降级试写时，默认派发一个 Opus 档位的 Agent 做一次正文写作测试。

执行规则：

1. 必须真的写正文，不能只做静态点评
2. 优先读取 `chapters/ch-{chapter_padded}-context.md`
3. 试写范围优先选择**最容易暴露问题的连续剧情段落**
4. 优先级顺序：
   - 中段转折 / 桥接段
   - 重要角色首次出场
   - 章末钩子
   - 如果以上都不明显，则试写章首切入
5. 测试稿保存到 `chapters/ch-{chapter_padded}-opus-test.md`
6. 反推报告保存到 `chapters/ch-{chapter_padded}-opus-report.md`

```text
调用一个 Opus 档位 Agent，执行一次正文写作测试。

输入文件：
1. chapters/ch-{chapter_padded}-outline.md
2. chapters/ch-{chapter_padded}-context.md（如存在）

任务：
1. 从大纲中选出最容易暴露问题的连续剧情段落，优先测试转折、桥接、首次重要出场或章末钩子
2. 基于这些剧情段落写 1000-1800 字测试正文
3. 将测试正文保存到 chapters/ch-{chapter_padded}-opus-test.md
4. 再基于本次试写过程，输出一份反推报告并保存到 chapters/ch-{chapter_padded}-opus-report.md

反推报告必须回答：
- 哪一段最难自然写开？为什么？
- 哪个转场需要补桥接动作、触发信息或情绪递进？
- 哪个角色的动机、关系或语言依据不足，导致台词发虚？
- 哪条信息给早了、给晚了、给密了，影响阅读推进？
- 哪些内容是你为了写顺而被迫脑补的？这些脑补应回填到大纲还是设定？
- 最终判定是：可直接写 / 需小修后写 / 需先重做大纲
```

如果测试暴露明显结构问题，先回到修订模式，不要直接鼓励用户进入 `/novel-write`。

## 大纲写作规则

所有大纲文本必须遵守：

### 第三人称精简剧情纲要

- ✅ “他踢翻桌子。茶杯碎了。凌绝尘没有动。” — 具体动作，不说目的
- ❌ “他愤怒地踢翻桌子，展现出内心的不满” — 解释动机 + 叙事评论
- ✅ “残魂说阵法是垃圾。苏寒染没理他。继续拧螺丝。” — 用动作和对话推进
- ❌ “通过残魂的吐槽，侧面展现了修真界阵法的落后” — 旁白式总结
- ✅ “苏寒染推门进殿。她先看向香炉，再看向凌绝尘。” — 第三人称转述
- ❌ “我推门进殿，先看向香炉。” — 第一人称，不符合大纲约定

### 禁止出现

- “展现出”“体现了”“暗示了”“为……埋下伏笔”这类叙事功能评论
- 大段心理分析
- 面向正文执行的碎片化字段
- 任何剧情点编号或执行表格式

## 关键原则

1. **先定节奏，再定任务，再做展开** — 本章先确认结构职责，再开始细化
2. **写给人看，不是写给 agent 看** — 大纲的第一目标是方便作者手改
3. **信息源有优先级** — future/ 与上一章 outline 优先于 `past.md`
4. **直接询问** — 不使用 AskUserQuestion，对话式沟通
5. **新规划模式文件只写一次** — 只有大纲成形后才写入文件
6. **修订模式差分更新** — 需要调整时先确认边界，再改已有大纲，不默认整章重写
7. **聚焦单章** — 全书级与分卷级规划由 `novel-bookplan` 负责
8. **章节只承担职责，不重复造骨架** — 不把单章写成另一个全书规划
9. **派生文件不阻断主流程** — context 失败时保留大纲成果并明确提示补跑
10. **修订后必须做联动检查** — 至少检查前章承接、本章职责、伏笔状态、下一章起手
11. **先试写，再放行** — 大纲确认后默认跑一次 Opus 正文测试，用真实写作暴露问题

**禁止**：

- 使用 AskUserQuestion 工具
- 未经讨论直接修改文件
- 一次性抛出多个问题
- 只问问题不给方案
- 未经用户确认就保存大纲
- 用低优先级文件覆盖高优先级信息
- 跳过本章节奏定位和章节任务卡
- 在大纲中写角色台词或语言风格细节（`novel-write` 的职责）
- 用户说“在原大纲上改”时仍强行走从零规划流程
- 修订局部时顺手把整章重写，导致用户无法追踪改动
