# 剧情规划

## 何时使用

- 需要落单章大纲、确认本章职责或修订已有章节大纲
- 需要在正式写正文前把结构问题先收敛掉

## 调用边界

- 负责逐章规划与单章修订，不负责全书级节奏重排
- Opus 全文试写验证大纲可写性，不替代 `novel-write`
- 内置规划期强制反思、爽点检查和反常规检查，尽量在大纲阶段拦住结构问题

## 目标

通过两种模式帮助用户完成单章工作：

- **新规划模式**：先定节奏职责、再定任务、最后写出可手改的双层大纲
- **修订模式**：先读取已有大纲、再界定变更边界、最后做差分修改
- **正文测试**：用户确认大纲后，默认调用一个 Opus agent 全文试写正文，再根据试写结果反推大纲问题

本技能只负责把这一章规划清楚，不负责把这一章切成写作单元。写作单元切分发生在 `novel-write` 阶段内部。

## 内置规划检查

`novel-plan` 直接负责规划期强制反思、爽点检查和反常规检查。主 skill 只保留执行节点和回退规则，详细检查标准和判断话术统一放在 `references/planning-checks.md`。

执行原则：

- 章节任务卡确认前，先做爽点检查，确认本章是否真的承担了该章该承担的爽点和钩子职责
- `小剧情点对齐卡` 确认前，先做剧情点级强制反思，判断哪些推进点删掉后主线仍然成立
- 正式大纲写入前，再做反常规检查，确认情节、角色反应、冲突升级和转折意外性不过于常规
- 如果检查结果已经指向结构层失效，就停在 `novel-plan` 内重修，不把问题留到正文阶段

## 使用方式

```bash
/novel-plan
/novel-plan --revise
```

## 新契约

### 你现在要产出的是什么

`chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` 必须是一份适合作者直接阅读和手改的章节简纲，而不是给写作 agent 看的执行表。

**必须包含**：

- 本章节奏定位
- 章节任务卡
- 起始状态
- 剧情思路卡
- 可写场景纲要
- 结束目标
- 伏笔处理

**不再输出**：

- 剧情点编号
- 情绪走向栏位
- 场景类型栏位
- 字数配额
- 面向写作 agent 的细碎执行抓手

### 正式大纲生成前必须先完成什么

- 先把选定走向拆成 3-8 个小剧情点
- 每个小剧情点都必须承担一次不可忽略的净推进
- 一个小剧情点通常只承载一个主目标、一次主冲突落地或一次明确状态变化
- 为每个小剧情点补齐 `Who / What / Why / Where / When / How`
- 先输出 `小剧情点对齐卡` 给用户确认
- 先确认对齐卡，再生成正式大纲
- `小剧情点对齐卡` 只用于澄清剧情，不写入 `大纲.md`

### 你现在不该做什么

- 不要把本章切成 10-20 个剧情点
- 不要把正文阶段才需要的切分结果写回 `大纲.md`
- 不要把 `小剧情点对齐卡` 当成最终大纲正文保存
- 不要为了“方便写作”牺牲“方便人工修改”

### 章节标题如何处理

- 章节标题默认不是必填即时决策
- 正式大纲草稿确认后，再单独处理章节标题
- 先给 2-3 个标题候选，也允许用户自己定标题
- 如果用户不想现在定标题，允许先记为“待定”
- 未经用户确认，不要擅自生成或写入章节标题

## 执行流程

### 前置：初始化检查

1. 检查 `memory/future/00-index.md` 和 `memory/future/10-book.md` 是否存在。不存在则建议先 `/novel-bookplan`。
2. **先读取 `CLAUDE.md` 的默认创作风格基线**：确认项目是否已经声明主角定位、装逼主驱动、情节取舍规则和默认爽点循环。后续章节任务卡、爽点检查和小剧情点展开都必须优先服从这份项目级约束。如果 `CLAUDE.md` 中包含"小说宪法"，宪法优先于默认规则；宪法中"作者人格"的叙事姿态、情感温度、幽默方式等维度直接影响场景设计的语气和详略判断。
3. **确定章节编号与所属卷**：从 `CLAUDE.md` 的「创作进度 → 当前章节」读取，或由用户指定（如 `/novel-plan 5`）。所属卷优先从 `memory/future/00-index.md`、对应卷卡或当前上下文判断；若无法定位 `volume_padded`，必须先提示用户补卷信息，不允许继续写入旧平铺路径。
4. **建立章节目录契约**：本章所有产物统一写入同一个章节目录：
   - 章节目录：`chapters/vol-{volume_padded}/ch-{chapter_padded}/`
   - 大纲：`chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md`
   - 上下文：`chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`
   - Opus 试写：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus试写.md`
   - Opus 报告：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`
   - 缺失设定：`chapters/vol-{volume_padded}/ch-{chapter_padded}/缺失设定.md`
5. **确定开头上下文包与停止边界**：
   - 开头固定按以下顺序读取，不默认铺开整个 `future/`，也不要直接扫描 `memory/entities/`
   - **共通读取顺序**：
     1. 先读取当前所在 `arc` 的剧情设定。优先读取 `memory/future/40-arcs/` 对应文件；如果当前 `arc` 尚未独立成文件，再回退到对应卷卡中的 `arc` 职责段
     2. 如果 arc 文件中包含**弧段详略策略**表格，提取当前章节的密度标记（高/中/低），后续在场景块和写作约束中继承
     2. 从当前 `arc` 文本里提取本章直接相关的角色、地点、物品、势力、概念；如需补读设定，必须先使用 `novel-knowledge` 搜索这些实体，再按命中结果读取对应设定文件，不要直接翻目录
     3. 读取 `memory/future/20-threads.md`，确认长期线程当前状态（哪些 Active / Pending / PaidOff）
     4. 读取 `memory/future/90-sync-tracker.md`，提取所有未回收伏笔及其来源章节、计划回收位置
     5. 读取上一章 `chapters/vol-{prev_volume_padded}/ch-{prev_padded}/正文.md`
     6. 再读取 `memory/past.md`
   - **修订模式额外输入**：
     - 在共通读取顺序前，先读取 `chapters/vol-{volume_padded}/ch-{current_padded}/大纲.md`
     - 再读取用户明确提出的修改目标
   - 读取到这里就停止；`memory/past.md` 只用于提取最近剧情摘要、当前状态和待处理伏笔，不要因为 `past.md` 中出现的设定名词继续展开读取对应设定文件
   - `20-threads.md` 和 `90-sync-tracker.md` 的作用是弥补 `past.md` 伏笔摘要的颗粒度不足：`past.md` 只提供概要，追踪表提供具体来源章节和计划回收位置；两者结合后，规划时不再需要回读更早章节的正文来追伏笔
   - `chapters/vol-{prev_volume_padded}/ch-{prev_padded}/上下文.md` 只供 `novel-write` 使用，不反向主导规划；修订模式下当前章 `上下文.md` 也只用于判断缓存是否失效

如果高优先级文件缺失，明确提示缺哪一个；不要擅自用低优先级信息替代。

### 前置：模式判定

- 用户要从零设计下一章，或当前章还没有 outline → **新规划模式**
- 用户明确说要在已有大纲上调整、修改、补桥接、改结尾、重写后半章 → **修订模式**
- 当前章已有 outline，且用户指出“这里太突兀 / 后半章不顺 / 收尾想改” → **修订模式**

如果进入修订模式，默认不回到从零发散；只有现有骨架失效，或用户明确要求大改时，才退回新规划模式。

## 第一段：定节奏

### 第一步：读取上下文

开头读取上下文时，先读取当前所在 `arc` 的剧情设定；如需补读角色、地点、物品、势力或概念，必须先使用 `novel-knowledge` 搜索当前 `arc` 中明确出现的相关设定，再读取 `memory/future/20-threads.md`（长期线程状态）和 `memory/future/90-sync-tracker.md`（未回收伏笔追踪），再读取上一章 `正文.md`，最后读取 `memory/past.md`。

读取到 `memory/past.md` 即停止，不要因为 `past.md` 中出现的设定名词继续展开读取对应设定文件。

然后输出当前进度、已完成章节、待处理伏笔、当前卷信息，并整理起始状态：

```text
📖 规划上下文

当前进度：
   卷：Vol-01（初入修真）
   已完成章节：第 10 章
   当前境界：炼气三层
   当前位置：凌霄宗外门

最近完成：
   第10章：演武场击败王师兄

待处理伏笔：
   - 空灵根的秘密（Vol-03）
   - 神秘势力的关注（Ch 20）

【起始状态】
   • 境界：炼气三层 | 位置：凌霄宗外门
   • 资源：灵石 50
   • 关系：敌对-王师兄，友好-李师姐
   • 近期：演武场切磋，展现不俗实力
```

**衔接检查**：

读取上一章 `正文.md` 的章末动作、最后一个有效钩子和已落地状态变化，与本章起始状态对比。发现不一致时明确标注缺哪一步过渡。

**章节伏笔衔接检查**：

优先从 `memory/future/90-sync-tracker.md`（未回收伏笔追踪表）和 `memory/future/20-threads.md`（长期线程状态）中筛选本章仍需承接的伏笔和线程；再用 `memory/past.md` 最近剧情摘要做补充。追踪表提供具体来源章节和计划回收位置，past.md 提供概要，两者结合后覆盖更完整。不要为了追伏笔顺着 `past.md` 再去扩读对应设定文件。

如果卷级节奏信息不足，明确提示：

```text
⚠️ 卷级节奏信息不足

我可以继续规划单章，但这会降低和整卷结构的贴合度。
更稳妥的做法：先回 /novel-bookplan 补该卷的节奏卡，再回来继续。
```

### 第二步：生成本章节奏定位

在章节任务卡之前，先输出本章在全书/分卷中的结构位置。

```text
📍 本章节奏定位

所属卷：Vol-02
主母型：Institutionalized
副母型：Fool Triumphant
卷内位置：中前段
承接的全书 beat：Fun and Games 区间

本章主职责：
1. 兑现这一卷的题材承诺
2. 推进主角在组织内部的站位变化
3. 为卷中点反转埋预热

本章不负责：
1. 提前揭开终极真相
2. 提前打完本卷 Boss
3. 一次性回收所有上一章伏笔

本章密度定位：[高（展开）/ 中（标准）/ 低（压缩）] — 继承自 arc 详略策略
```

如果本章只是承上启下，也要直说，不要硬编成大转折章：

```text
本章类型：过桥章
结构价值：把角色从 A 状态送到 B 状态，同时保留钩子
额外说明：过桥章也要说明在为下一次装逼兑现做什么蓄势
```

## 第二段：定任务

### 第一步：生成章节任务卡

基于当前 `arc` 的章节职责、通过 `novel-knowledge` 命中的直接相关设定、上一章 `正文.md` 的已落地状态和 `memory/past.md` 中的待处理伏笔，先提出一个可执行的章节任务卡，再让用户确认。

在输出任务卡前，先额外自问四个问题：

- 本章主任务是否服务主角装逼链条
- 本章最核心的爽点兑现是什么
- 哪个角色或局面承担压制、轻视、对比或被打脸作用
- 哪些内容与主角装逼无关，可以延期

```text
📌 章节任务卡

本章主任务：进入五丰镇并完成第一次有效对接

必须推进的线：
1. 与周华顺建立联系
2. 展示夜市与鲁班坊体系
3. 让叶芷恒完成第一次有效出场

允许延期的线：
1. 空灵根暗线继续压后
2. 通讯阵更深层异常先不揭开

建议收尾点：苏寒染说“我要验验”
```

如果用户要改，优先改任务卡，不要直接跳到展开层。

### 第二步：确认本章范围和收尾点

在任务卡确认后，再确认本章收尾点和大致篇幅：

```text
📐 章节范围建议

建议收尾点：苏寒染说“我要验验”
字数预估：2800-3600 字

这个收尾点可以吗？还是想提前/延后？
```

## 第三段：做展开

### 第一步：一句话走向

以下流程默认用于**新规划模式**。修订模式只有在现有骨架已失效，或用户明确要求重做后半章/整章时，才回到这一段。

先检查 `memory/future/40-arcs/` 或对应卷卡中，本章是否已有明确事件规划或章节职责段。

**有明确规划** → 先复述职责，再确认是否沿用。  
**没有明确规划** → 给出 10 个可能走向，让用户选 1-2 个方向，不使用 AskUserQuestion。

用户选择或确认后，只围绕这三个维度补问，且一次只问一个：

1. 本章涉及哪些角色
2. 本章结束时角色状态会怎样变化
3. 本章需要承接前章的哪些悬念/线索

### 第二步：场景化方案

先选择叙事模板，再基于选定走向和叙事模板设计 3-6 个场景化方案。场景化方案只用于帮助你想清楚结构，不直接写入大纲。

不要把工具箱应用说明、规划检查记录或过程性旁注暴露给用户。

### 第三步：拆分小剧情点并补齐 5W1H

场景化方案确定后，不要立刻写双层大纲，先把本章推进链拆开。

**执行规则**：

1. 从选定方案中拆成 3-8 个小剧情点
2. 每个小剧情点都必须承担一次不可忽略的净推进
3. 一个小剧情点通常只承载一个主目标、一次主冲突落地或一次明确状态变化
4. 每个小剧情点都必须补齐 `Who / What / Why / Where / When / How`
5. 如果某个维度不明确，只围绕当前缺口补问，且一次只问一个
6. `How` 要写清触发方式、推进动作或冲突落地方式，不能只写抽象结果
7. 这些小剧情点只用于澄清，不等于 `novel-write` 的写作单元

**颗粒度判定**：

- 场景不等于小剧情点
- 如果一个点里包含两次独立转折、跨地点切换或跨时间跳步，就应该拆开
- 如果只是同一次推进里的连续小动作，不要硬拆成多个点
- 小剧情点名称优先写成“谁为了什么做了什么，并因此发生了哪次变化”，不要只写空泛场景名

**输出格式**：

```text
🧩 小剧情点对齐卡

1. [剧情点名称]
- Who：
- What：
- Why：
- Where：
- When：
- How：
```

**确认规则**：

- 先确认对齐卡，再生成正式大纲
- 对齐卡里只要有一个关键维度含混，就继续澄清，不跳过
- 对齐卡是对话中的中间产物，不写入 `大纲.md`
- 如果用户只改其中几个点，就只更新那些点，再继续后续步骤

### 第四步：设定完整度与命名一致性检查

在 `小剧情点对齐卡` 确认后，再检查本章涉及的关键设定是否齐全。

**执行方式**：

1. 从选定方案中提取涉及的角色、地点、物品/概念
2. 检查 `memory/entities/` 下的对应设定文件是否存在且包含必填字段
3. 检查本章中计划使用的称呼是否能映射到已有实体

**检查范围**：

- 角色：基本信息、外貌描述、核心性格、语言风格、动机、修为境界、人物关系
- 地点：基本描述、环境/氛围、特点
- 物品/概念：基本描述、功能/效果、限制/条件

**输出完整度报告**：

```text
📖 设定完整度报告

【苏寒染】主角 | ✅ 完整
【凌绝尘】配角 | ⚠️ 语言风格缺失
【周华顺】配角 | ❌ 无设定文件
【五丰镇夜市】地点 | ⚠️ 仅有基本描述
```

**输出命名一致性报告**：

```text
🪪 命名一致性检查

【叶芷恒】canonical name 已存在
- 本章称呼：叶姑娘 | ✅ 可用（首次见面，视角合理）
- 本章称呼：叶芷恒 | ⚠️ 需确认是否已经知名
- 本章称呼：那少女 | ✅ 可用（短暂过渡）
```

如果发现设定缺失，立即停止，写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/缺失设定.md`，并提示用户去另一个 session 用 `/novel-discuss` 补全。

### 第五步：生成双层大纲

这是本技能最关键的输出步骤。

你要写的是一份**双层大纲**，不是执行表，也不是把剧情思路压扁成几段顺滑摘要。

这里使用的是**已经确认过 `5W1H` 的小剧情点链条**，但最终要拆成两层：

- `剧情思路卡`：解释这章为什么这样排
- `可写场景纲要`：告诉正文这一章怎么落

#### 第一层：剧情思路卡

`剧情思路卡` 必须固定包含以下 5 项：

- `本章核心兑现`
- `主角状态变化`
- `核心冲突链`
- `信息释放策略`
- `本章边界`

写法要求：

- 这层写的是作者意图，不是摘要，不是口号
- `核心冲突链` 优先写成“因为 X，主角必须做 Y；做 Y 时撞上 Z；因此局面变成 W”
- `信息释放策略` 要明确“本章明给什么 / 本章压住什么”
- `本章边界` 要直接写这章不负责什么，防止大纲贪多

#### 第二层：可写场景纲要

`可写场景纲要` 建议写成 3-6 个场景块。每个场景块必须固定包含：

- `密度`：继承本章密度定位，可在场景间微调（展开章的某场景也可能标低，压缩章的某场景也可能标高）
- `场景目标`
- `触发事件`
- `冲突落地`
- `转折点`
- `落点变化`
- `细节锚点`

写法要求：

- 场景块写的是可落文骨架，不是结果摘要
- `触发事件` 必须写清是谁做了什么，场面才真的启动
- `冲突落地` 必须写成动作、问答、试探、打断、压价、验货、插手等可见行为
- `转折点` 必须指出哪一下真正改变了局势
- `落点变化` 必须落到关系、判断、风险、目标或信息占有中的至少一项
- `细节锚点` 每场保留 1-3 个，供正文阶段抓住硬信息，不允许只写空泛氛围

#### 推荐结构

```text
## 基本信息
## 起始状态
## 章节任务卡
## 剧情思路卡
## 可写场景纲要
## 结束目标
## 伏笔处理
```

#### 示例

```text
## 剧情思路卡

- 本章核心兑现：第一次有效接触不是“搭上线”，而是确认这条线本身带坑
- 主角状态变化：苏寒染从“先看热闹、顺手摸门路”变成“主动盯住这条线，准备反验”
- 核心冲突链：因为苏寒染要尽快摸清夜市门道，他接受周华顺带路；周华顺借带路试探他是否懂行；因此双方从单纯交易转成互相试底
- 信息释放策略：本章明给“夜市不干净、周华顺在设套”；本章压住“周华顺背后到底是谁”
- 本章边界：不提前揭背后主使，不提前打爆整条线

## 可写场景纲要

### 场景 1：先看热闹，再看出门道
- 密度：中（标准）← 继承本章定位
- 场景目标：让苏寒染先摸清夜市表层秩序，再意识到这里不只是热闹
- 触发事件：他刚入夜市，就看见几处摊位的价码和看货顺序不对劲
- 冲突落地：他不直接发问，而是先看别人怎么买、怎么被带路
- 转折点：一个外行买家被人几句话带偏，让苏寒染意识到这里的规矩本身就是筛人
- 落点变化：苏寒染从“逛”切到“验”，警惕性被正式拉起来
- 细节锚点：
  - 他先看成交顺序，不先看货
  - 同一件东西，对熟客和生客报价不同
  - 第一处异常不是假货，而是话术
```

5. 标题确认不应反过来改动已确认的大纲骨架；如标题暴露结构理解偏差，再回到大纲层修正

- 如果标题尚未确认，在 `基本信息` 中写 `章节标题：待定`

## 修订模式（基于已有大纲调整）

### 第一步：读取当前章并生成修订简报

先读取当前章节的已有大纲，提炼：

- 本章当前职责
- 起始状态和结束目标
- 现有剧情骨架
- 已安排的伏笔回收/埋设
- 当前版本最可能出问题的地方

输出格式：

```text
🩺 当前章修订简报

当前骨架：
1. 前半章先完成入场和信息对接
2. 中段发生被骗/试探
3. 结尾落在主角决定反击

建议保留：
1. 夜市初见的世界观展示
2. 周华顺的第一次有效接触

待解决问题：
1. 中段转折过快
2. 叶芷恒出场偏功能化
3. 结尾钩子不够利落
```

### 第二步：提出修订策略

不要让用户从零描述怎么改。先给 2-3 个修订策略，并推荐一个最稳的。

### 第三步：确认变更边界

确认时把边界说具体：

```text
📏 本次修订边界

保留：
1. 开场到夜市入场
2. 周华顺第一次接触

删除/弱化：
1. 原中段里的直接冲突

新增：
1. 一个被骗前的试探场景

收尾：
1. 从“先记下了”改成“我要验验”
```

### 第四步：重做受影响小剧情点的 5W1H

修订模式不默认整章回到从零规划，而是先缩小影响范围。

**执行规则**：

1. 先判断本次修改会影响哪些小剧情点
2. 只对新增、改写或受波及的小剧情点重做 `5W1H`
3. 未受影响的小剧情点默认沿用，不要求整章重新确认
4. 如果局部修改已经破坏整章推进链，才退回新规划模式

如需重新对齐，仍使用 `小剧情点对齐卡` 展示受影响部分，但只展示需要重做的点。

### 第五步：生成修订清单并更新已有大纲

更新规则：

1. 优先保留未受影响的段落，不为了统一风格整章重写
2. 改的是局部，就只动局部
3. 如果结束目标、伏笔处理、涉及角色已被改动，要同步更新对应板块
4. 如果只是局部桥接或收尾调整，优先只改受影响的场景块
5. 如果本章核心兑现、核心冲突链或信息释放策略被改动，要同步更新 `剧情思路卡`

### 第六步：输出联动检查

修订完成后，必须追加一轮联动检查：

```text
🔎 修订后联动检查

上一章承接：
   ✅ 起始状态仍能接上

本章职责：
   ✅ 仍然承担 Fun and Games 的兑现

伏笔状态：
   ⚠️ 原计划本章回收的[伏笔A]已延期到下一章

下一章承接：
   ⚠️ 如果下一章仍按旧计划开头，需要把第一场改成“验货/对质”
```

如果已有 `chapters/vol-{volume_padded}/ch-{current_padded}/上下文.md`，且本次修订影响了以下任一项，视为缓存可能失效：

- 出场角色变化
- 关键地点变化
- 伏笔状态变化
- 本章结束目标变化

## 用户确认后执行的后续步骤

用户确认大纲后，按顺序执行以下两步（参考 `references/output.md`）：

### 第一步：尝试生成上下文缓存

调用 context-collector agent（`subagent_type: "context-collector"`），生成 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`。

```text
为第 {chapter} 章生成上下文缓存。

大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md

请根据大纲中的角色、地点、物品，收集完整的上下文信息。
将输出保存到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md。
```

如果 context 生成失败，明确提示：

```text
⚠️ 大纲已完成，但 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md 尚未生成。
后续写正文前，请重新执行 /novel-plan 或单独补跑 context 收集。
```

### 第二步：执行 Opus 正文写作测试

在 context 生成完成后，或确认本次只能用 outline 降级试写时，默认派发一个 Opus 档位的 Agent 做一次**全文正文写作测试**。

执行规则：

1. 必须真的写正文，不能只做静态点评
2. 优先读取 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`
3. 如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`，必须先完整读取其中缓存内容，再开始试写
4. 试写时必须优先复用已有上下文缓存，不要绕过缓存重新临时组织一份 context
5. **全文试写**：必须覆盖整章全部场景块，不允许只挑部分场景块试写；如果当前章节还是旧版结构，则按完整剧情纲要从头写到尾
6. 字数目标与正文阶段对齐：2500-4000 字
7. 测试稿保存到 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus试写.md`
8. 反推报告保存到 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`
9. `Opus报告.md` 仍保存完整报告，不因为对话层改成逐题确认而省略问题细节
10. Opus报告.md 仍保存完整报告，供后续修订和复盘读取
11. 面向用户反馈时，先把报告里的可行动问题整理成待确认队列，再逐个确认
12. 一次只问一个问题，优先拿最高优先级、最影响是否放行写作的那个问题
13. 每个问题都必须附 2-3 个修法选项，不能只丢问题不给路
14. 当前问题未确认前，不进入下一个问题
15. 如果当前问题一经确认就足以判断必须先回到 `/novel-plan --revise`，就在这一题后分流，不再继续展开次级问题
16. 待确认问题未清空前，不直接鼓励用户进入 `/novel-write`
17. 如果某个问题属于 agent 可以按既有约束直接局部修补、无需用户拍板，则不要为了凑数强行提问

```text
调用一个 Opus 档位 Agent，执行一次全文正文写作测试。

输入文件：
1. chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md
2. chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md（如存在）

任务：
1. 如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`，必须先完整读取其中缓存内容，再开始试写
2. 试写时必须优先复用已有上下文缓存，不要绕过缓存重新临时组织一份 context
3. 全文试写：必须覆盖 `可写场景纲要` 中的全部场景块，不允许只挑部分场景；如果当前章节只有旧版 `第三人称精简剧情纲要`，则按完整纲要从头写到尾
4. 字数目标 2500-4000 字，与正文阶段对齐
5. 将测试正文保存到 chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus试写.md
6. 再基于本次全文试写过程，输出一份反推报告并保存到 chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md

反推报告必须回答：
- 哪一段最难自然写开？为什么？
- 哪个转场需要补桥接动作、触发信息或情绪递进？
- 哪个角色的动机、关系或语言依据不足，导致台词发虚？
- 哪条信息给早了、给晚了、给密了，影响阅读推进？
- 哪些内容是你为了写顺而被迫脑补的？这些脑补应回填到大纲还是设定？
- 最终判定是：可直接写 / 需小修后写 / 需先重做大纲
```

拿到 `Opus报告.md` 后，向用户反馈时使用如下收口规则：

```text
🩺 Opus 试写反馈

先不要把完整问题清单一次性抛给用户。
先在内部整理待确认队列，再逐个确认。

当前轮只处理 1 个问题，格式固定为：
1. 当前最高优先级问题是什么
2. 这个问题为什么会阻断或影响写作
3. 提供 2-3 个修法选项，并给出推荐项
4. 等用户确认后，再决定是否进入下一个问题
```

如果测试暴露明显结构问题，先回到修订模式；待确认问题未清空前，不直接鼓励用户进入 `/novel-write`。

## 大纲写作规则

所有大纲文本必须遵守：

### 剧情思路卡

- 要写“为什么这样排”，不要写“这一段用来展现……”
- `核心冲突链` 必须写成因果链，不允许只写结论
- `信息释放策略` 必须明确“本章明给什么 / 本章压住什么”

### 可写场景纲要

- ✅ “周华顺先抛熟人价，再故意递错货，看苏寒染会不会先摸边角。” — 触发和冲突都可见
- ❌ “双方开始试探，关系发生变化。” — 结果先行，没有动作
- ✅ “苏寒染只指出一处做旧痕迹，却故意不说全，逼周华顺先补话。” — 转折点清楚
- ❌ “他意识到对方并不简单。” — 摘要腔，没有交代如何意识到

### 大纲阶段抗 AI 味检查

- 生成时主动清理摘要腔、功能说明腔、空泛转场和结果先行
- 如果一句话里只剩“他意识到 / 他确认了 / 关系发生变化”，就回去补触发、冲突和转折
- 不要用“随后”“经过一番周折”偷渡场景切换，先写清触发动作

### 禁止出现

- “展现出”“体现了”“暗示了”“为……埋下伏笔”这类叙事功能评论
- “双方开始博弈”“气氛愈发紧张”“关系发生变化”这类空泛结果句
- 大段心理分析
- 面向正文执行的碎片化字段
- 任何剧情点编号或执行表格式

## 关键原则

1. **先定节奏，再定任务，再做展开** — 本章先确认结构职责，再开始细化
2. **写给人看，不是写给 agent 看** — 大纲的第一目标是方便作者手改
3. **开头上下文有固定边界** — 先读当前 `arc`，再用 `novel-knowledge` 定向补设定，再读 `20-threads.md` 和 `90-sync-tracker.md` 追踪伏笔线索，再读上一章 `正文.md` 和 `past.md`，到此停止
4. **直接询问** — 不使用 AskUserQuestion，对话式沟通
5. **新规划模式文件只写一次** — 只有大纲成形后才写入文件
6. **修订模式差分更新** — 需要调整时先确认边界，再改已有大纲，不默认整章重写
7. **聚焦单章** — 全书级与分卷级规划由 `novel-bookplan` 负责
8. **章节只承担职责，不重复造骨架** — 不把单章写成另一个全书规划
9. **派生文件不阻断主流程** — context 失败时保留大纲成果并明确提示补跑
10. **修订后必须做联动检查** — 至少检查前章承接、本章职责、伏笔状态、下一章起手
11. **先全文试写，再放行** — 大纲确认后默认跑一次 Opus 全文正文测试，用真实写作暴露问题

**禁止**：

- 使用 AskUserQuestion 工具
- 未经讨论直接修改文件
- 一次性抛出多个问题
- 只问问题不给方案
- 在 Opus 试写后把整份问题清单直接复述给用户
- 当前问题还没确认就继续追问下一个问题
- 待确认问题还没清空就提前放行 `/novel-write`
