# 输出规格

## 角色设定完整度验证

**用户审阅通过大纲后，自动扫描每个出场角色的设定完整度。**

大纲不需要包含角色的语言风格细节，但需要确保每个涉及角色的设定文件存在且包含必填字段，否则 `novel-write` 无法稳定创作台词。

**扫描规则**：

| 字段 | 标记 | protagonist | supporting/antagonist | 说明 |
|------|------|:-----------:|:--------------------:|------|
| 基本信息 | 🔴 | 必填 | 必填 | name, type, age, identity |
| 外貌描述 | 🔴 | 必填 | 必填 | 体型/面容/装束，至少一句话 |
| 核心性格 | 🔴 | 必填 | 必填 | 3-5 个核心性格特点 |
| 语言风格 | 🔴 | 必填 | 必填 | `novel-write` 创作台词的直接依据 |
| 动机 | 🔴 | 必填 | 必填 | 角色的核心驱动力 |
| 修为境界 | 🔴 | 必填 | 必填 | 当前修为等级 |
| 人物关系 | 🔴 | 必填 | 必填 | 至少有一条关系记录 |
| 记忆点（视觉+行为+反差） | 🟡 | 必填 | 建议填写 | 角色辨识度的核心来源 |
| 语言记忆点 | 🟡 | 必填 | 建议填写 | 口头禅或标志性句式 |
| 在剧情中的作用 | 🟡 | 必填 | 可选 | 角色的叙事功能 |
| 行事风格 | 🟡 | 必填 | 可选 | 行为模式 |

**扫描执行方式**：

1. 从大纲「涉及角色」中提取所有角色名
2. 逐一读取 `memory/entities/characters/{角色名}.md`
3. 检查每个🔴字段的章节标题是否存在且内容非空
4. 输出完整度报告

**输出格式**：

```text
📖 角色设定完整度报告

【苏寒染】主角 | 完整度：85%
  ✅ 基本信息 / 外貌描述 / 核心性格 / 语言风格 / 动机 / 修为境界 / 人物关系
  ⚠️ 记忆点（视觉）— 未填写视觉记忆点
  ⚠️ 记忆点（语言）— 未填写语言记忆点
  ❌ 无缺失必填字段

【凌绝尘】配角 | 完整度：60%
  ✅ 基本信息 / 核心性格 / 修为境界
  ⚠️ 外貌描述 — 内容为空
  ❌ 语言风格 — 缺失
  ❌ 动机 — 缺失
  ❌ 人物关系 — 缺失
```

## 上下文缓存生成

**在保存大纲的同时，生成章节级上下文缓存文件，供 `novel-write` 直接复用。**

生成方式：调用 context-collector agent（subagent_type: "context-collector"），prompt 如下：

```text
为第 {chapter} 章生成上下文缓存。

大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md

请根据大纲中的角色、地点、物品，收集完整的上下文信息。
将输出保存到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md。
```

**context-collector 会自动**：

1. 分析大纲中的涉及角色、地点、物品
2. 调用知识图谱搜索相关设定
3. 读取设定文件
4. 读取 `past.md` 最近剧情摘要
5. 读取相关 future/ 线程与卷卡
6. 输出上下文缓存文件到 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`

## Opus 正文写作测试

**在用户确认大纲后，默认执行一次 Opus 正文写作测试。**

这一步不是点评大纲，而是让一个 Opus agent 根据当前大纲真实写一段正文，然后基于实际写作中的卡顿、脑补和节奏问题反推大纲缺陷。

**输出文件**：

- 测试正文：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus试写.md`
- 反推报告：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`

**测试规则**：

1. 优先读取 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`
2. 如果 context 缺失，允许降级为 outline-only 试写，但报告顶部必须写 `置信度：低`
3. 试写范围优先选取最容易暴露问题的连续剧情段落
4. 测试正文必须是真正文，不少于 1000 字
5. 报告必须基于“写出来之后的卡点”反推问题，不能只做抽象评论

**判定等级**：

- `可直接写`：Opus 能基本顺滑写出正文，结构和承接没有明显硬伤
- `需小修后写`：主线没问题，但桥接、信息分配、角色声线等局部问题需要补强
- `需先重做大纲`：Opus 试写时需要大量脑补，说明结构本身不可直接落文

## 大纲保存格式

将完整的章节简纲保存到 `chapters/vol-xx/ch-xxxx/大纲.md`，格式如下：

```markdown
# 第 X 章大纲：[标题]

## 基本信息
- 本章类型：[推进章 / 过桥章 / 转折章]
- 承接 beat：[Save the Cat 或卷内职责]

## 起始状态
- 境界：
- 位置：
- 资源：
- 关系：
- 最近关键变化：

## 结束目标
1. [目标1]
2. [目标2]

## 章节任务卡
- 本章主任务：
- 必须推进的线：
- 允许延期的线：
- 建议收尾点：

## 涉及角色

### 苏寒染
- 本章作用：[作用描述]

### [其他角色]
- 本章作用：[作用描述]

## 第三人称精简剧情纲要

[使用 3-6 段短段落，写清本章从开端到收尾的推进过程。]
[必须是第三人称。]
[必须方便作者直接手改。]
[不要做剧情点切分。]

## 伏笔处理
- 本章回收：[伏笔] → [处理方式]
- 本章延期：[伏笔] → [延期到哪里]
- 本章新埋设：[伏笔] → [预计后续位置]
```

## 精简剧情纲要写法

**目标**：让作者一眼看懂这一章发生了什么、为什么这样推进、结尾落在哪里，并且可以直接替换、删减、挪动段落。

**必须做到**：

- 用第三人称转述
- 写动作、冲突、信息变化和关系变化
- 允许自然分段，但不做“剧情点1/2/3”
- 每段都是正在发生的剧情，不是对剧情的评论

**禁止出现**：

- 面向写作 agent 的执行字段
- 情绪走向、场景类型、字数配额
- 叙事功能评论，如“这一段用来展现……”
- 第一人称或第二人称

## 修订模式输出

当 `novel-plan` 是基于已有 outline 做调整时，除了更新 `chapters/vol-xx/ch-xxxx/大纲.md`，还要在对话中输出一份**修订摘要**，方便用户快速审阅本次改动。

**输出格式**：

```text
🧩 本次修订摘要

保留：
1. 开场入城和夜市入场不变

改写：
1. 中段试探增强信息差

新增：
1. 一个被骗前的短桥接

删除/弱化：
1. 重复解释段

联动影响：
1. 本章结尾改为“我要验验”
2. 下一章开头动作需要同步调整
```

**修订模式最少检查项**：

1. 起始状态是否仍与上一章结尾一致
2. 结束目标是否仍符合本章节奏职责
3. 伏笔计划是否需要延期 / 改回收位置
4. 涉及角色列表是否需要增删
5. 现有 `上下文.md` 是否失效
