# 输出规格

## 角色设定完整度验证

**用户审阅通过大纲后，自动扫描每个出场角色的设定完整度。**

大纲不需要包含角色的语言风格细节，但需要确保每个涉及角色的设定文件存在且包含必填字段，否则 `novel-write` 无法稳定创作台词。

**扫描规则**：

| 字段 | 标记 | protagonist | supporting/antagonist | 说明 |
|------|------|:-----------:|:--------------------:|------|
| 基本信息 | 🔴 | 必填 | 必填 | name, type, age, identity |
| 外貌描述 | 🔴 | 必填 | 必填 | 体型/面容/装束，至少一句话 |
| 核心性格 | 🔴 | 必填 | 必填 | 3-5 个核心性格特点 |
| 语言风格 | 🔴 | 必填 | 必填 | `novel-write` 创作台词的直接依据 |
| 动机 | 🔴 | 必填 | 必填 | 角色的核心驱动力 |
| 修为境界 | 🔴 | 必填 | 必填 | 当前修为等级 |
| 人物关系 | 🔴 | 必填 | 必填 | 至少有一条关系记录 |
| 记忆点（视觉+行为+反差） | 🟡 | 必填 | 建议填写 | 角色辨识度的核心来源 |
| 语言记忆点 | 🟡 | 必填 | 建议填写 | 口头禅或标志性句式 |
| 在剧情中的作用 | 🟡 | 必填 | 可选 | 角色的叙事功能 |
| 行事风格 | 🟡 | 必填 | 可选 | 行为模式 |

**扫描执行方式**：

1. 从大纲「涉及角色」中提取所有角色名
2. 逐一读取 `memory/entities/characters/{角色名}.md`
3. 检查每个🔴字段的章节标题是否存在且内容非空
4. 输出完整度报告

**输出格式**：

```text
📖 角色设定完整度报告

【苏寒染】主角 | 完整度：85%
  ✅ 基本信息 / 外貌描述 / 核心性格 / 语言风格 / 动机 / 修为境界 / 人物关系
  ⚠️ 记忆点（视觉）— 未填写视觉记忆点
  ⚠️ 记忆点（语言）— 未填写语言记忆点
  ❌ 无缺失必填字段

【凌绝尘】配角 | 完整度：60%
  ✅ 基本信息 / 核心性格 / 修为境界
  ⚠️ 外貌描述 — 内容为空
  ❌ 语言风格 — 缺失
  ❌ 动机 — 缺失
  ❌ 人物关系 — 缺失
```

## 上下文缓存生成

**在保存大纲的同时，生成章节级上下文缓存文件，供 `novel-write` 直接复用。**

生成方式：调用 context-collector agent（subagent_type: "context-collector"），prompt 如下：

```text
为第 {chapter} 章生成上下文缓存。

大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md

请根据大纲中的角色、地点、物品，收集完整的上下文信息。
将输出保存到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md。
```

**context-collector 会自动**：

1. 分析大纲中的涉及角色、地点、物品
2. 调用知识图谱搜索相关设定
3. 读取设定文件
4. 读取 `past.md` 最近剧情摘要
5. 读取相关 future/ 线程与卷卡
6. 输出上下文缓存文件到 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`

## Opus 正文写作测试

**在用户确认大纲后，默认执行一次 Opus 正文写作测试。**

这一步不是点评大纲，而是让一个 Opus agent 根据当前大纲真实写一段正文，然后基于实际写作中的卡顿、脑补和节奏问题反推大纲缺陷。

**输出文件**：

- 测试正文：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus试写.md`
- 反推报告：`chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`

**用户反馈方式**：

- `Opus报告.md` 的问题必须完整保存到 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`
- 面向用户时不要整包转述报告，而是把问题整理为待确认队列后逐个确认
- 一次只处理 1 个问题
- 每个问题都要附 2-3 个修法选项，并给出推荐方案
- 当前问题没确认前，不进入下一个问题
- 只要待确认问题还没清空，就不要直接鼓励用户进入 `/novel-write`
- 如果当前问题已经足以判定应先回到 `/novel-plan --revise`，就在该问题确认后立即分流

**测试规则**：

1. 优先读取 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`
2. 如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md`，必须先完整读取其中缓存内容，再开始试写
3. 试写时必须优先复用已有上下文缓存，不要绕过缓存重新临时组织一份 context
4. 如果 context 缺失，允许降级为 outline-only 试写，但报告顶部必须写 `置信度：低`
5. 试写范围优先选取最容易暴露问题的场景块；如果当前章节还是旧版结构，再回退为连续剧情段落
6. 测试正文必须是真正文，不少于 1000 字
7. 报告必须基于“写出来之后的卡点”反推问题，不能只做抽象评论

**判定等级**：

- `可直接写`：Opus 能基本顺滑写出正文，结构和承接没有明显硬伤
- `需小修后写`：主线没问题，但桥接、信息分配、角色声线等局部问题需要补强
- `需先重做大纲`：Opus 试写时需要大量脑补，说明结构本身不可直接落文

## 双层大纲写法

**目标**：让作者一眼看懂这一章为什么这样排，并且让 `novel-write` 直接拿到可落文的场景骨架。

**必须做到**：

- `剧情思路卡` 说清本章核心兑现、状态变化、冲突链、信息释放和边界
- `可写场景纲要` 写清场景目标、触发、冲突、转折、落点和细节锚点
- 每个场景块必须标明密度（高/中/低），继承自本章密度定位，可在场景间微调
- 密度的含义：高密度场景下游写作时给更大字数空间、要求感官层次和角色反应写足；低密度场景下游写作时只保留动作和信息推进，砍环境描写和配角心理
- 触发和冲突必须可见，不能只写结果
- 允许自然分场，但不要把场景块膨胀成半章正文

**禁止出现**：

- 面向写作 agent 的执行字段
- 情绪走向、场景类型、字数配额
- 叙事功能评论，如“这一段用来展现……”
- 只写“他意识到”“关系发生变化”这种摘要腔结果句
- 第一人称或第二人称

## 修订模式输出

当 `novel-plan` 是基于已有 outline 做调整时，除了更新 `chapters/vol-xx/ch-xxxx/大纲.md`，还要在对话中输出一份**修订摘要**，方便用户快速审阅本次改动。

**输出格式**：

```text
🧩 本次修订摘要

保留：
1. 开场入城和夜市入场不变

改写：
1. 中段试探增强信息差

新增：
1. 一个被骗前的短桥接

删除/弱化：
1. 重复解释段

联动影响：
1. 本章结尾改为“我要验验”
2. 下一章开头动作需要同步调整
```

**修订模式最少检查项**：

1. 起始状态是否仍与上一章结尾一致
2. 结束目标是否仍符合本章节奏职责
3. 伏笔计划是否需要延期 / 改回收位置
4. 涉及角色列表是否需要增删
5. 现有 `上下文.md` 是否失效
