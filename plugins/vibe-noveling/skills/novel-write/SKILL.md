# 章节创作

## 核心理念

章节写作负责三件事：

1. **把大纲变成血肉** — 在章节骨架上补环境、感官、动作链和台词
2. **把执行切分变成可核对骨架** — 先在正文阶段固定细切为 20 个剧情点；20 个剧情点不回写 outline，但要显性保留在所有稿件里，方便人工核对
3. **提供稳定的逐点择优** — 同一章先生成多种笔法，再按剧情点逐个选择最合适的版本合并成最终稿

## 第一步：获取上下文

### 1.1 检查上下文缓存

首先检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md` 是否存在且有效。

缓存有效性判断：

- 文件存在 → 读取缓存头的 `生成时间`
- 检查相关设定文件的修改时间是否晚于缓存生成时间：
  - 出场角色的设定文件（`memory/entities/characters/`）
  - `memory/worldbuilding.md`
  - `memory/_graph.json`
- 所有文件都未修改 → **缓存有效**
- 有文件被修改 → **缓存失效，重新收集**

### 1.2 根据缓存状态选择路径

**缓存有效**：

```text
📖 使用上下文缓存

检测到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md（缓存有效）
生成时间：{timestamp}
设定文件无变更 → 直接复用
```

**缓存不存在或已失效**：

```text
📖 缓存不可用

{缓存不存在 / 缓存已失效：角色设定文件已变更}

调用 context-collector 重新收集上下文...
```

调用方式：

```text
Agent 工具，subagent_type: context-collector
prompt: 为第 X 章收集上下文，大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md
```

### 1.3 设定缺失处理

如果 context-collector 报告设定缺失，必须先补全设定，再继续写作。

禁止：

- 忽略设定缺失
- 自己编造设定
- 假设“后续再补”

## 第二步：读取大纲

从 `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` 读取本章大纲。

**大纲关键认知**：

- 上游优先给的是 `剧情思路卡` + `可写场景纲要`
- `剧情思路卡` 用来约束本章核心兑现、冲突链和信息释放边界
- `可写场景纲要` 用来承接触发事件、冲突落地、转折点与落点变化
- 若章节尚未升级，上游给的是**第三人称精简剧情纲要**，按兼容路径降级读取
- 大纲的目标是让作者易读、易改，不是给正文执行用的分镜表
- 你要从中提取推进链，而不是要求上游已经把它切成剧情点

**重点读取内容**：

- 基本信息
- 起始状态
- 剧情思路卡
- 可写场景纲要
- 结束目标
- 章节任务卡
- 第三人称精简剧情纲要（仅旧版兼容路径必读）
- 伏笔处理

**禁止**：

- 擅自改变章节收尾目标
- 擅自改掉关键推进链
- 忽略 `剧情思路卡` 的本章边界与核心兑现约束
- 把大纲重新解释成另一个不同的大纲

## 第二步 A：正文阶段细化剧情点

在正式写作前，先在正文阶段固定细切为 20 个剧情点。

### 2A.1 切分目标

优先从 `可写场景纲要` 细切出恰好 20 个**剧情点**；若仅存在旧版结构，则从 `第三人称精简剧情纲要` 细切。每个剧情点只需要回答五件事：

1. 当前剧情点的标题是什么
2. 这一点从什么动作或状态开始
3. 这一点的核心冲突或信息推进是什么
4. 这一点的关系变化或情绪压力是什么
5. 这一点把压力推到哪里结束

### 2A.2 剧情点规则

- 剧情点数量固定为 20，不允许多于或少于 20 个
- 每个剧情点都必须承担一次明确的动作推进、信息释放、关系位移或压力变化
- 单个剧情点可以短，但不能空转，不能拿一串连续小动作硬凑成无意义切片
- 20 个剧情点不回写 outline
- 所有中间稿和最终稿都必须显性保留剧情点标题
- 中间稿标题格式统一为 `【剧情点01：标题】`
- 最终稿标题格式统一为 `【剧情点01：标题｜来源：肘子】`
- 同一个剧情点内部可以自由扩写台词、动作和细节
- 如果切分后发现必须新增关键事件、删除关键事件或重排章节收尾，停止并回到 `novel-plan --revise`

### 2A.3 密度继承与传导

剧情点从所属场景块继承密度标记（高/中/低），并传导至写作约束。

**继承规则**：

- 从大纲的可写场景纲要中读取每个场景块的密度字段
- 每个剧情点继承其所属场景的密度
- 如果场景未标密度，继承基本信息中的本章密度定位
- 如果本章密度也未标注，默认为中（标准）

**密度对剧情点的影响**：

- **高密度剧情点**：字数空间更大（200-400字/点），必须包含感官层次（视觉+听觉+触觉）、在场角色反应、主角内心冲击
- **中密度剧情点**：标准处理（120-200字/点）
- **低密度剧情点**：压缩处理（50-100字/点），只保留动作推进和信息释放，禁止环境描写、配角心理、微动作、过场废话

**传导方式**：

密度标记写入 `剧情点.md`，每个剧情点条目追加密度字段，格式如下：

```markdown
### 剧情点 03：[标题]
- 密度：高（展开）← 继承自场景 2
- 起始状态：
- 核心推进：
- 关系变化：
- 结束压力：
```

### 2A.3 何时必须退回 `novel-plan`

以下情况必须停止正文阶段硬修：

- 需要新增关键剧情段
- 需要删除整段核心推进
- 需要重排关键剧情顺序
- 需要改动章节收尾目标
- 需要把伏笔处理位置整体后挪或前移

### 2A.4 剧情点落盘

将细切完成的 20 个剧情点写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/剧情点.md`。

格式要求：

- 每个剧情点包含：编号、标题、起始状态、核心推进、关系变化、结束压力
- 此文件是所有 writer subagent 和合并 agent 的共享输入
- 落盘后不再通过 prompt 内联传递

## 第二步 B：读取 Opus 测试报告（如存在）

如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`，必须在正式写正文前先读取它。

这份报告的作用有两个：

1. **放行判定**：这一章现在能不能直接进入正文创作
2. **写作约束**：正式写作时必须补出、必须避免、必须压缩的点

### 2B.1 放行规则

**如果不存在报告**：

```text
📎 未找到 Opus 测试报告

本次按 outline + context 直接写作。
风险：不会自动继承上游试写发现的问题点。
```

**如果报告判定为 `需先重做大纲`**：

立即停止，不进入风格写作阶段，直接回到 `/novel-plan --revise`。

**如果报告判定为 `需小修后写`**：

只有当这些问题能被转成**局部写作约束**时，才允许继续写。否则回到 `novel-plan --revise`。

### 2B.2 约束提炼方式

传给写作 agent 的应该是短的“写作约束摘要”，而不是整份诊断报告。摘要应包含：

- 必须补出的桥接动作
- 必须压缩的信息段
- 需要加固的角色声线
- 禁止重犯的问题

### 2B.3 约束落盘

将提炼的"写作约束摘要"写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/写作约束.md`。
若不存在 Opus 报告，写入"无额外约束"。

密度约束同时追加到 `写作约束.md`，格式如下：

```markdown
## 详略约束

高密度剧情点（{编号列表}）：
  - 字数空间：200-400字/点
  - 必须包含：感官层次、在场角色反应、主角内心冲击
  - 禁止压缩，宁可写多

低密度剧情点（{编号列表}）：
  - 字数空间：50-100字/点
  - 只保留：动作推进 + 信息释放
  - 禁止：环境描写、配角心理、微动作、过场废话

中密度剧情点（{编号列表}）：
  - 字数空间：120-200字/点
  - 标准处理
```

## 第二步 C：合并配置

在剧情点细切完成后、正式写作前，确认本章的合并配置。

### 2C.1 检查已有配置

检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/合并配置.md` 是否存在。

**如果已存在**：读取并确认配置内容，进入第三步。

### 2C.2 协助创建配置（无已有配置时）

如果 `合并配置.md` 不存在，**必须使用 `AskUserQuestion` 工具**向用户询问本章的合并偏好。

调用示例：

```text
AskUserQuestion({
  questions: [{
    header: "合并配置",
    multiSelect: false,
    options: [
      { label: "采纳建议", description: "使用基于本章大纲自动生成的合并偏好建议" },
      { label: "手动调整", description: "我想自己指定风格映射和基调偏好" },
      { label: "使用默认规则", description: "跳过配置，完全使用 references/merge-rules.md 默认规则" }
    ],
    question: "根据本章大纲，我建议以下合并偏好：\n[基于剧情点特征给出建议，例如：本章以战斗场景为主（剧情点 5-12），建议「正面冲击」优先会说话的肘子；本章布局试探较多（剧情点 13-16），建议「布局交锋」优先大仲马；整体基调：硬朗打斗]\n\n是否采纳以上建议？"
  }]
})
```

如果用户选择"手动调整"，再追问具体偏好（风格映射、基调倾向、需要避免的风格等）。

用户确认后，将配置写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/合并配置.md`。

配置文件格式：

```markdown
# 合并配置

## 风格偏好
| 剧情点特征 | 优先风格 | 理由 |
|-----------|---------|------|
| 战斗、正面冲击 | 会说话的肘子 | 本章以演武打斗为主 |
| 试探、布局 | 大仲马 | 本章暗线交锋多 |

## 全局规则覆盖
<!-- 如需追加额外合并约束或替换默认风格优先规则，在此声明 -->
<!-- 无则留空 -->

## 章节基调
- 本章气质倾向：打斗硬朗
- 本章应避免的风格：（无）
```

### 2C.3 使用默认规则（用户跳过配置时）

如果用户表示"用默认规则"或跳过此步骤，不创建 `合并配置.md`，后续合并阶段完全使用 `references/merge-rules.md` 中的默认规则。

## 第三步：双风格写作准备

默认创作以下 2 种风格版本：

1. **会说话的肘子（底稿）**— 现代快节奏战斗、临场决断、干净利落的冲突推进
2. **大仲马（高光）**— 精密布局、戏剧交锋和信息差

中间稿全部保留到本章目录 `chapters/vol-{volume_padded}/ch-{chapter_padded}/`，供回看、局部抽取和后续复用。

### 3.1 并发数确认

在正式调用 writer subagent 前，**必须使用 `AskUserQuestion` 工具**向用户确认并发数：

```text
AskUserQuestion({
  questions: [{
    header: "并发数",
    multiSelect: false,
    options: [
      { label: "1 并发（串行）(Recommended)", description: "逐个调用，最稳定" },
      { label: "2 并发", description: "同时写 2 个风格" }
    ],
    question: "本章需要创作 2 个风格版本（肘子底稿 + 大仲马高光）。请选择并发数。"
  }]
})
```

用户选择后记录并发数，供第四步使用。如果用户跳过，默认 1 并发（串行）。

## 第四步：调用 writer subagent + 校对

按以下顺序调用 writer subagent：

1. 会说话的肘子 → `writer-zhouzi`（底稿）
2. 大仲马 → `writer-dazhongma`（高光）

### 4.1 并发调度

根据用户在第三步确认的并发数，采用不同调度策略：

- **1 并发（串行）**：每次只调用 1 个 writer subagent，前一个完成后再调用下一个
- **2 并发**：同时调用 2 个 writer subagent
- 2 个风格版本都完成基础校对后，再进入后续一致性检查和自动合并

调用时使用 Agent 工具，并将对应风格路由到固定 subagent_type：

```text
会说话的肘子 → `subagent_type: writer-zhouzi`
大仲马 → `subagent_type: writer-dazhongma`
```

writer subagent 负责稳定文风人格、风格自检和通用写作规则；`novel-write` 主 skill 只需通过 prompt 传入章节动态任务和必读文件路径，不再内联展开上下文、大纲、剧情点或写作约束。

### 动态写作任务 Prompt 模板

```text
请按以下要求创作第 {chapter} 章的正文。

## 文件写入要求

你必须将最终完成的正文写入：
`chapters/vol-{volume_padded}/ch-{chapter_padded}/{style_name}.md`

## 基本信息
- 章节：第 {chapter} 章
- 写作风格：{style_name}
- 字数目标：3000-5000 字

## 必读文件（按顺序读取）

1. `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md` — 角色设定、前情、伏笔
2. `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` — 本章大纲
3. `chapters/vol-{volume_padded}/ch-{chapter_padded}/剧情点.md` — 20 个剧情点（含密度标记）
4. `chapters/vol-{volume_padded}/ch-{chapter_padded}/写作约束.md` — Opus 报告约束摘要 + 详略约束

读取规则：
- 上下文缓存文件直接原样读取，不要另写抽象总结
- 只有在缓存长度超限时，才允许压缩为紧凑摘录
- 压缩时必须保留角色语言风格、地点约束、伏笔状态和写作注意事项
- 剧情点数量必须恰好 20 个，中间稿必须完整保留 `【剧情点01：标题】` 这类标题
- 不得把 20 个剧情点压成无标题连续正文

## 正文写作一致性说明

- writer 按照各自 agent 文档中的风格人格、通用写作规则和章节动态任务落文
- 重点保证剧情推进清晰、文风稳定、角色声音准确、章节节奏自然
- 只要不破坏 20 个剧情点结构、不违背大纲与写作约束，具体句式和段落组织由 writer 自然发挥

## 思维模式

【思维模式要求】在你的思考过程（<think>标签内）中，请遵守以下规则：
1. 禁止使用圆括号包裹内心独白，例如"（心想：……）"或"(内心OS：……)"，所有分析内容直接陈述即可
2. 禁止以角色第一人称描写内心活动，例如"我心想""我觉得""我暗自"等，请用分析性语言替代
3. 思考内容应聚焦于剧情走向分析和回复内容规划，不要在思考中进行角色扮演式的内心戏表演
```

## 第五步：按剧情点逐点择优合并最终稿

调用subagent，自动合并中间稿，生成最终稿。

### 自动合并 Prompt 模板

```text
你是一个正文合并 agent。请将以下 2 个风格版本按 20 个剧情点逐点择优合并为第 {chapter} 章最终稿。
不要整章一次性融合，也不要跨风格拼句；每个剧情点只能选择 1 个主来源版本。

## 必读文件
- chapters/vol-{volume_padded}/ch-{chapter_padded}/剧情点.md — 20 个剧情点摘要
- chapters/vol-{volume_padded}/ch-{chapter_padded}/合并配置.md — 章节级合并配置（如存在）

## 输入文件
- chapters/vol-{volume_padded}/ch-{chapter_padded}/会说话的肘子.md
- chapters/vol-{volume_padded}/ch-{chapter_padded}/大仲马.md

## 输出文件
- chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md

## 合并原则

### 不可覆盖的硬约束
1. 按 20 个剧情点顺序逐点择优合并
2. 每次只处理当前剧情点在 2 个风格版本中的对应内容
3. 每个剧情点只选 1 个主来源版本
4. 只允许对选中的剧情点做必要的局部修字，不从其他版本拼句
5. 当前剧情点标题必须标记来源风格，格式为 `【剧情点07：雨巷试探｜大仲马】`
6. 最终稿必须完整保留 20 个剧情点标题，供用户人工核对

### 规则来源与优先级
1. 检查是否存在 `合并配置.md`
2. 如果存在章节合并配置：
   - 风格偏好：章节配置的「风格偏好」表优先于宪法推导的偏好
   - 全局约束：章节配置的「全局规则覆盖」追加到默认约束之后
   - 章节基调：作为模糊判断时的收束依据
3. 如果不存在章节合并配置，则凭直觉选择一个风格。

### 合并后自检
- 是否完整保留了 20 个剧情点标题
- 是否每个剧情点都只标了 1 个来源风格
```

## 第六步：合并后最终稿一致性校验

对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 先做一轮最终稿一致性校验。

执行顺序：

1. 先调用 `consistency-guard`，重点检查：
   - 合并过程中是否引入新设定冲突
   - 不同版本拼接后是否出现称谓、时间线、认知不一致
   - 来源切换后角色语气、信息掌握范围和场景因果是否断裂
2. 若发现问题，只允许做局部修补；不得借一致性修补名义重做结构
3. 一致性校验通过后，才能进入下一步正文 AI 味检测

## 第七步：合并后正文 AI 味检测与文风复检

在最终稿一致性校验通过后，对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 执行 AI 味检测。本步由主 session 直接执行，不委托 agent。

执行方式：

1. 读取 `{SKILL_DIR}/references/ai-smell-checklist.md` 作为唯一检测清单来源
2. 读取正文文件
3. 按清单中 10 个检测项逐项复检正文，每一轮检测仅检查一个检测项，并修复一个检测项暴露的所有问题。
   1. 发现问题后做定点修正：定位到具体段落，按检测项建议的方式修改，修改后验证不引入新问题
   2. 修正完成后写回正文
4. 输出检测报告
5. AI 味检测通过后，继续执行第九步润色；润色完成后继续第十步仿人工增强

## 第八步：AI 味清理后的正文润色

在 AI 味检测与文风复检完成后，再对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 执行一轮独立润色。

执行要求：

1. 物品描写：避免只写“普通、简单、破旧、发亮”这类直白词，优先替换成更有质感、更有层次的华丽表达
   物品描写不要先报物品名称，再用短句补特征；要直接把它写成“是什么样的东西”
2. 环境描写：优先补出氛围、光泽、纹理、层次和压迫感，让场景显得更精美、更高级
3. 人物描写：优先使用更有风骨、气势、锋芒和仪态感的词汇，不把人物描写写得发平
4. 行为描写：把角色的动作描写，改写成带武侠风味的动作表达
5. 润色后的句子仍必须服务主角推进，不得因为追求华丽词汇而让动作发虚、信息发糊

## 第九步：仿人工增强

在润色完成后、交付前，对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 执行一轮仿人工增强。

### 执行方式

本步由主 session 直接执行，不委托 agent。

#### 第一阶段：虚词密度统计

1. 读取正文，统计全文总字数（不含标点和空格）
2. 统计以下全部 7 类虚词的出现次数：
   - **动态助词**：了、着、过
   - **结构助词**：的、地、得
   - **语气词**：么、吧、呢、啊、嘛、哦、哈、呀
   - **介词**：在、从、向、把、被、对、为、给、用、比、按
   - **副词**：就、还、都、也、又、才、再、更、最、很、真、好、太、已经、正在、刚刚
   - **判断动词**：是、有
   - **连词**：而、但、却、且、或
3. 计算每类虚词的密度（次/100字）

#### 第二阶段：偏差计算

与手工精修基准对比。容忍区间为基准密度的 ±10%（即基准 × 0.9 ~ 基准 × 1.1 范围内不处理）：

| 虚词类别 | 基准密度(/100字) | 内部分配             |
| -------- | ---------------- | -------------------- |
| 动态助词 | 3.15             | 了:着:过 ≈ 60:26:14 |
| 结构助词 | 3.12             | 的:地:得 ≈ 68:27:4  |
| 介词     | 2.68             | 在/从/向/把/被 为主  |
| 副词     | 2.44             | 就/还/都/也 为主     |
| 判断动词 | 1.22             | 是/有                |
| 语气词   | 0.51             | 么/吧/呢 为主        |
| 连词     | 0.27             | 而/但/却 为主        |

对每个子类分别计算缺口和超额：

**缺口计算**（低于基准，需补插）：

- 缺口 = max(0, 基准 × 0.9 - 当前密度) × 总字数 / 100
- 缺口取整数，向上取整
- 当前密度 ≥ 基准 × 0.9 的，不做补插

**超额计算**（高于基准，需缩减）：

- 超额 = max(0, 当前密度 - 基准 × 1.1) × 总字数 / 100
- 超额取整数，向上取整
- 当前密度 ≤ 基准 × 1.1 的，不做缩减

**额外操作**：删句号（目标：约每 8-10 个句号删 1 个，用 SSoT 决定删哪些）

#### 第三阶段：SSoT 双向校准

1. 在心里生成一个复杂随机字符串（至少 30 字符），不暴露给用户
2. 根据缺口/超额数量，用随机字符串的不同片段做 rolling hash，为每个需要操作的位置随机选择一个句子序号
3. 确保同一个句子最多只被选中一次（避免堆叠）
4. 按缺口和超额分别执行：

   ##### 补插操作（填补缺口）

   **简单插入类**（不改句式，直接在合适位置插入虚词）：


   - **加「了」**：在最近动词后或分句末插入，表示完成态或变化
   - **加「着」**：在最近动词后插入，表示持续态
   - **加「过」**：在最近动词后插入，表示经历态
   - **加「的」**：在名词短语后或分句末插入
   - **加「地」**：在动词前的状语位置插入（XX地做某事）
   - **加「得」**：在动词后的补语位置插入（做得XX）
   - **加语气词**：在句末标点前插入，增加口语感
   - **加副词**：在动词或形容词前的合适位置插入

   **句式改写类**（允许调整句式来自然补入虚词，但必须保持原意不变）：

   - **补介词**：改写句式引入介词结构（如「他走进房间」→「他从门外走进了房间」）
   - **补判断动词**：改写句式引入判断结构（如「他很厉害」→「他是个厉害角色」）
   - **补连词**：在分句间插入连词（如「他赢了，没人不服」→「他赢了，但没人不服」）

   ##### 缩减操作（消除超额）

   **安全删除类**（直接删，不影响语法）：

   - **删副词**：删除冗余的程度副词（很、真、太、非常等）
   - **删语气词**：删除句末多余的语气词（哦、嘛、哈、呀等）
   - **删冗余「了」**：删除完成态标记中非必需的「了」（如「他站了起来」→「他站起」）
   - **删冗余「的」**：删除名词短语末尾非必需的「的」（如「他的脸上带着开心的笑容」→「他脸上带着笑容」）

   **需判断删除类**（检查上下文，非必需则删）：

   - **删结构助词「地」**：删除非必需的状语标记
   - **删结构助词「得」**：删除非必需的补语标记（简短句式可合并）
   - **删动态助词「着」「过」**：持续态/经历态非必需时删除

   **句式改写类**（通过改写句式来减少虚词用量）：

   - **减判断动词**：把「是/有」句改为直接描述（如「他是个勇敢的人」→「他很勇敢」）
   - **减介词**：简化介词结构（如「他从门外走进了房间」→「他走进房间」）
   - **减连词**：去掉分句间连词，用逗号自然衔接（如「他赢了，但没人不服」→「他赢了，没人不服」）

   ##### 其他操作

   - **删句号**：删除句末「。」，让相邻句子自然流接（只删「。」，不删「！」「？」「……」）

#### 第四阶段：验证与写回

1. 重新统计虚词密度，确认校准效果

   **补插验证**：

   - 对每个存在缺口的子类，确认密度已回升到基准 × 0.9 以上
   - 如果仍低于基准 × 0.9，再做一轮补插

   **缩减验证**：

   - 对每个存在超额的子类，确认密度已降至基准 × 1.1 以下
   - 如果仍高于基准 × 1.1，再做一轮缩减

   **过度校准回退**：

   - 若某个类别补插过度（当前密度 > 基准 × 1.5），回退多余的插入
   - 若某个类别缩减过度（当前密度 < 基准 × 0.5），回退多余的删除
2. 处理完成后写回 `正文.md`

### 跳过规则

以下内容不参与仿人工增强：

- 剧情点标题行（`【剧情点...】` 格式）
- 空行
- 纯分隔行（`***`）

### 执行边界

- 简单虚词做定点插入/删除，不改句式
- 介词/判断动词/连词的补入或缩减允许改写句式，但必须保持原意不变、不改变事实信息和动作因果
- 不对同一段落内的句子做合并或拆分
- 如果某类虚词的缺口和超额均为 0，直接跳过该类别
- 缩减操作不得导致语义变化或信息丢失

- 哪些剧情点使用了大仲马版本（高光段落）


## 关键原则

1. **忠于大纲** — 在双层大纲约束内自由发挥台词和场景细节；若当前章节只有旧版 `第三人称精简剧情纲要`，也只在其范围内落文，不擅自改变剧情方向
2. **剧情点就是核对骨架** — 正文阶段必须固定细切为 20 个剧情点，并在所有稿件里显性保留
3. **风格分明** — 每种风格有明显差异，不是微调
4. **质量保证** — 每个版本和最终稿都经过校对和一致性检查
5. **自动合并有明确依据** — 最终稿按剧情点逐点择优，每个剧情点只选 1 个主来源版本，不做句群拼贴；合并规则支持章节级配置覆盖
6. **先看报告，再写正文** — `opus-report` 是放行条件和约束来源
7. **结构问题不上正文阶段硬修** — 需要新增/删除/重排关键剧情段时，回到 `novel-plan --revise`
8. **冗余描述在正文阶段清理** — 不把“水字数”主要寄希望于上游试写，正式成稿后必须再删一轮
9. **最终稿向项目正文气质收束** — 保留能提气的犀利观察、冷面反讽和精准吐槽，但不放任成段子腔或 POV 外调侃
10. **AI 味清理后再做润色** — AI 味检测由主 session 直接执行；检测完成后做高级感描写和武侠风动作润色，且不改剧情事实与因果
11. **润色后做仿人工增强** — 润色完成后，统计正文全部 7 类虚词密度并与手工精修基准对比，对低于基准的虚词类别用 SSoT 做定向补插，对超出基准的虚词类别用 SSoT 做定向缩减（均允许改写句式），使正文虚词分布接近手工精修水平
12. **返修与同步独立执行** — 正文交付后，用户运行 `/novel-revise` 进行返修，返修完成后运行 `/novel-sync` 进行同步；`novel-write` 不再包含返修和同步步骤
