# 章节创作

## 目标

根据 `novel-plan` 生成的第三人称精简剧情纲要，并结合上游 Opus 测试报告提炼出的写作约束，默认创作全部 4 种写作风格的正文版本。4 个版本不是一次性并发派发，而是按固定顺序逐个开启写作 Agent，避免同一时刻同时发出 4 个请求。每个版本经过校对后保留为中间稿。最终稿合并也不是整章一次性融合，而是按内部写作单元顺序逐个合并，再输出为连续章节正文 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`，用户确认满意后进入同步流程。

**核心特性**：

- 默认创作全部 4 种写作风格
- 按风格逐个开启写作 Agent，避免一次性派发 4 个请求
- 在正文阶段内部切分写作单元
- 按内部写作单元顺序逐个合并最终稿
- 自动合并生成最终稿
- 保留中间稿供回看和局部抽取
- 最终稿输出为连续章节正文

## 核心理念

章节写作负责三件事：

1. **把简纲变成血肉** — 在章节骨架上补环境、感官、动作链和台词
2. **把内部切分藏在执行层** — 先在正文阶段内部切分写作单元，再按风格逐个开启写作 Agent；写作单元只供写作阶段使用，不回写 outline
3. **提供稳定的风格融合** — 同一章先生成多种笔法，再按写作单元顺序逐个合并成最终稿

上游大纲已经决定了：

- 本章职责
- 起始状态
- 结束目标
- 核心推进链
- 收尾点

正文写作负责：

- 把这些变成可读的连续章节
- 根据角色设定文件创作差异化台词
- 在不改大纲骨架的前提下，自己解决冗余、节奏和落文问题

## 内置 AI 味检测

`novel-write` 直接负责 20 个 AI 味检测项。主 skill 只保留执行时机和边界，完整清单和最终稿专属矫正规则统一放在 `references/ai-smell-checklist.md`。

执行原则：

- 中间稿写作时先按清单做轻量自检，但不提前统一抹平 4 个中间稿的风格差异
- 最终稿复检时按完整清单逐项复检，并强制执行最终稿文风矫正
- 水字数检查在正文阶段只处理句子、段落、场景内部铺陈级冗余，不负责删除整个剧情点或重排结构
- 如果检测结果显示必须改动结构层推进，停止正文硬修，回到 `novel-plan --revise`

## 使用方式

```bash
/novel-write 01
/novel-write 02
/novel-write next
```

       │  提炼“写作约束摘要”

       │  对每个版本调用 consistency-guard

## 第一步：获取上下文（优先使用缓存）

### 1.1 检查上下文缓存

首先检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md` 是否存在且有效。

缓存有效性判断：

- 文件存在 → 读取缓存头的 `生成时间`
- 检查相关设定文件的修改时间是否晚于缓存生成时间：
  - 出场角色的设定文件（`memory/entities/characters/`）
  - `memory/worldbuilding.md`
  - `memory/_graph.json`
- 所有文件都未修改 → **缓存有效**
- 有文件被修改 → **缓存失效，重新收集**

### 1.2 根据缓存状态选择路径

**缓存有效**：

```text
📖 使用上下文缓存

检测到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md（缓存有效）
生成时间：{timestamp}
设定文件无变更 → 直接复用
```

**缓存不存在或已失效**：

```text
📖 缓存不可用

{缓存不存在 / 缓存已失效：角色设定文件已变更}

调用 context-collector 重新收集上下文...
```

调用方式：

```text
Agent 工具，subagent_type: context-collector
prompt: 为第 X 章收集上下文，大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md
```

### 1.3 设定缺失处理

如果 context-collector 报告设定缺失，必须先补全设定，再继续写作。

禁止：

- 忽略设定缺失
- 自己编造设定
- 假设“后续再补”

## 第二步：读取大纲

从 `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` 读取本章的简纲。

**大纲关键认知**：

- 上游给的是**第三人称精简剧情纲要**
- 大纲的目标是让作者易读、易改，不是给正文执行用的分镜表
- 你要从中提取推进链，而不是要求上游已经把它切成剧情点

**重点读取内容**：

- 基本信息
- 起始状态
- 结束目标
- 章节任务卡
- 第三人称精简剧情纲要
- 伏笔处理

**禁止**：

- 擅自改变章节收尾目标
- 擅自改掉关键推进链
- 把简纲重新解释成另一个不同的大纲

## 第二步 A：正文阶段内部切分

在正式写作前，先在正文阶段内部切分写作单元。

### 2A.1 切分目标

从精简剧情纲要中提取 6-12 个**写作单元**。每个写作单元只需要回答四件事：

1. 这一段从什么动作或状态开始
2. 这一段的核心冲突或信息推进是什么
3. 这一段的关系变化或情绪压力是什么
4. 这一段把压力推到哪里结束

### 2A.2 写作单元规则

- 写作单元只供写作阶段使用，不回写 outline
- 写作单元不要求用户审阅
- 写作单元是执行拆分，不是新的用户可见格式
- 同一个写作单元内部可以自由扩写台词、动作和细节
- 如果切分后发现必须新增关键事件、删除关键事件或重排章节收尾，停止并回到 `novel-plan --revise`

### 2A.3 何时必须退回 `novel-plan`

以下情况必须停止正文阶段硬修：

- 需要新增关键剧情段
- 需要删除整段核心推进
- 需要重排关键剧情顺序
- 需要改动章节收尾目标
- 需要把伏笔处理位置整体后挪或前移

## 第二步 B：读取 Opus 测试报告（如存在）

如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`，必须在正式写正文前先读取它。

这份报告的作用有两个：

1. **放行判定**：这一章现在能不能直接进入正文创作
2. **写作约束**：正式写作时必须补出、必须避免、必须压缩的点

### 2B.1 放行规则

**如果不存在报告**：

```text
📎 未找到 Opus 测试报告

本次按 outline + context 直接写作。
风险：不会自动继承上游试写发现的问题点。
```

**如果报告判定为 `需先重做大纲`**：

立即停止，不进入风格写作阶段，直接回到 `/novel-plan --revise`。

**如果报告判定为 `需小修后写`**：

只有当这些问题能被转成**局部写作约束**时，才允许继续写。否则回到 `novel-plan --revise`。

### 2B.2 约束提炼方式

传给写作 agent 的应该是短的“写作约束摘要”，而不是整份诊断报告。摘要应包含：

- 必须补出的桥接动作
- 必须压缩的信息段
- 需要加固的角色声线
- 禁止重犯的问题

## 第三步：默认全风格写作准备

默认创作以下 4 种风格版本：

1. **海明威风（极简硬派）**
2. **卖报小郎君风（轻悬疑群像）**
3. **火星引力风（暴力美学+情感拉满）**
4. **大仲马风（精密布局与戏剧交锋）**

中间稿全部保留到本章目录 `chapters/vol-{volume_padded}/ch-{chapter_padded}/`，供回看、局部抽取和后续复用。

## 第四步：逐个开启 Agent + 校对

按以下固定顺序逐个开启 `general-purpose` 写作 Agent：

1. 海明威
2. 卖报小郎君
3. 火星引力
4. 大仲马

启动规则：

- 每次只开启一个新的写作 Agent
- 前一个 Agent 成功启动并接收完任务后，再开启下一个
- 不要一次性同时派发 4 个请求，避免请求失败
- 4 个风格版本都完成基础校对后，再进入后续一致性检查和自动合并

`chapters/vol-{volume_padded}/ch-{chapter_padded}/{style_name}.md`

## 基本信息
- 章节：第 {chapter} 章
- 写作风格：{style_name}
- 字数目标：2500-4000 字

## 上下文摘要
{context_content}

## 大纲内容
{outline_content}

## 内部写作单元摘要
{writing_units_summary}

说明：
- 这些写作单元来自正文阶段内部切分
- 写作单元只供执行，不是用户要看的格式
- 禁止把写作单元标题直接暴露到最终正文中

## Opus 测试约束摘要
{opus_constraints_summary}

## 通用写作要求

### 节奏控制
- 短句为主，长句点缀，段落不超过 4 行
- 紧张时一句一行，舒缓时允许稍长句
- 能删的字全删

### 句式规则
- 禁止使用“不是……，而是……”
- 对话要有刺，不写客气对白
- 越重要的事话说得越少
- 角色声音必须区分开

### 视角规则
- 紧贴主角有限第三人称
- 主角不知道的，读者不能知道
- 用感官和动作代替大段心理解释

### 输出格式
- 最终稿输出为连续章节正文
- 使用自然段落和必要的场景换行
- 不使用“剧情点1 / 剧情点2”标题
- 不暴露任何内部写作单元标记

### 结构边界
- 不新增关键剧情段
- 不删除关键推进
- 不重排章节收尾
- 如果必须这样做才能写通，停止并回到 novel-plan --revise

### 成稿后自检
1. 是否仍是连续章节正文
2. 是否出现了总结式、概述式转述
3. 是否重新犯了 Opus 报告里提醒的问题
4. 是否出现多余过渡或水字数
5. 是否有角色台词失真

### 冗余描述清理
如果删掉一句、一段或一组连续描写后，不影响任何主线因果、信息推进、角色关系变化、冲突升级或章末钩子，就删。
```

### 风格指令（注入到 Agent Prompt）

**海明威风（极简硬派）**

- 冰山理论，只写露出来的部分
- 动词优先，少修辞
- 对话越短越好
- 心理不直写，用动作和沉默代替

**卖报小郎君风（轻悬疑群像）**

- 节奏轻快但持续给线索
- 群像互动鲜活，每个人都有态度
- 信息尽量藏在对话、插话和动作里
- 章末必须留钩子

**火星引力风（暴力美学+情感拉满）**

- 关键动作要有正面冲击力
- 战斗和情绪段都要有临场感
- 情感不靠抒情，靠压强和失控边缘

**大仲马风（精密布局与戏剧交锋）**

- 重布局、交锋、信息差
- 对话里要有心机和试探
- 冲突不是纯爆发，而是层层逼近

## 第五步：一致性检查

对每个中间稿调用 `consistency-guard`。重点检查：

- 角色称谓是否漂移
- 时间线是否冲突
- 主角认知是否越界
- 合并前各稿是否已经引入额外设定

如发现问题：定位到具体段落做局部修改，修改后只重查被修改部分。一个版本修改 3 轮仍不通过，再展示给用户手动处理。

## 第六步：按写作单元逐个合并最终稿

在所有风格版本通过基础校对和一致性检查后，新增“自动合并”步骤，生成最终稿：

`chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`

自动合并不是把整章一次性糅成一个版本，而是按内部写作单元顺序逐个合并。

合并执行方式：

- 按内部写作单元顺序逐个合并
- 每次只处理当前写作单元在 4 个风格版本中的对应内容
- 当前写作单元合并完成后，再进入下一个写作单元
- 所有写作单元合并完成后，再自然衔接为连续章节正文

自动合并必须遵循以下固定规则：

1. **当前写作单元先定主骨架**：优先以卖报小郎君风承接当前单元的节奏、线索组织、群像协同和钩子推进
2. **风景描写优先使用海明威风**：在当前写作单元内优先吸收简洁、具体、少修辞的环境句群
3. **战斗描写优先使用大仲马风**：在当前写作单元内优先吸收布局感、攻防交锋、信息差和戏剧性推进
4. **火星引力风只做局部补强**：在当前写作单元内补正面冲击、情绪临场感和动作张力，不替换主骨架
5. **避免过多比喻**：单段最多一个明显修辞，能直写就直写
6. **出现纯第三人称概述时必须替换**：优先从当前写作单元对应的其他版本中抽取正面描写、动作链和直接对话替换
7. **最终稿输出为连续章节正文**：允许场景换行，不暴露内部切分结果

2. 每次只处理当前写作单元在 4 个风格版本中的对应内容
3. 当前写作单元合并完成后，再进入下一个写作单元
4. 当前写作单元优先以卖报小郎君版本为主骨架
5. 当前写作单元的风景描写优先取海明威版本
6. 当前写作单元的战斗描写优先取大仲马版本
7. 火星引力版本只做当前写作单元的局部补强
8. 删除拼接感明显、概述味太重的句群
9. 最终稿输出为连续章节正文，不暴露内部写作单元

## 合并后自检
- 是否保持章节节奏稳定
- 是否减少概述式转述
- 是否出现称谓漂移
- 是否仍是连续章节正文
- 是否在每个写作单元之间保持自然衔接
```

## 第七步：最终稿复检

对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 再做一轮检查，不能只默认相信合并结果。

检查顺序：

1. 按 `references/ai-smell-checklist.md` 中的 20 个 AI 味检测项逐项复检，重点检查：
   - AI 味是否回流
   - 高频警示词
   - 明喻使用限制
   - 重复率控制
   - 声音描写限制
   - 无意义细节禁止
   - 对话是否重新变得解释化
   - 合并后是否出现比喻堆砌
   - 是否重新出现总结式结尾
   - 执行最终稿文风矫正（检测项 16-19）
   - 这一步只在最终稿复检时强制执行
   - 不提前统一抹平 4 个中间稿的风格差异
2. 调用 `consistency-guard`，重点检查：
   - 合并过程中是否引入新设定冲突
   - 不同版本拼接后是否出现称谓、时间线、认知不一致
3. 若发现问题，只允许做局部修补；不得借合并名义重做结构

## 第八步：交付最终稿和中间稿

默认交付以下文件：

- 最终稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/海明威.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/卖报小郎君.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/火星引力.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/大仲马.md`

交付说明中应简短说明：

- 最终稿以哪个版本为骨架
- 哪些段落主要吸收了海明威或大仲马的表达
- 是否替换了概述味太重的弱段

## 第九步：字数统计与同步确认

对 4 个中间稿和最终稿分别统计字数。

**字数要求**：

- 每个中间稿：2500-4000 字
- 最终合并稿：允许在 2500-4500 字之间，只要节奏更紧、不水字数即可

**同步规则**：

- 自动合并完成后，不直接同步知识图谱
- 用户确认最终稿满意后，再执行：`/novel-sync chapter {chapter}`
- 用户确认语句示例：`确认 chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`

## 输出格式

每个中间稿和最终合并稿都必须输出为**连续章节正文**：

- 使用自然段落
- 允许必要的场景换行或 `***` 作为弱分隔
- 禁止使用内部写作单元标题
- 禁止暴露“剧情点1 / 剧情点2”这类执行层标记

│       ├── 火星引力.md
│       ├── 大仲马.md

### 文件写入时机

1. **创作阶段**：每个 Agent 自检通过后立即将正文写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/{style_name}.md`
2. **合并阶段**：按写作单元顺序读取全部中间稿的对应内容，逐个合并生成最终稿 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`
3. **交付阶段**：默认交付最终稿路径和各中间稿路径，必要时再展示局部正文
4. **同步阶段**：用户确认 `正文.md` 满意后，执行 `/novel-sync chapter {chapter}`
5. **保留阶段**：各风格中间稿默认保留，供回看、回滚和局部抽取

## 关键原则

1. **忠于大纲** — 在精简剧情纲要范围内自由发挥台词和场景细节，不擅自改变剧情方向
2. **先内部切分，再外部交付** — 写作单元属于执行层，不属于用户可见层
3. **风格分明** — 每种风格有明显差异，不是微调
4. **质量保证** — 每个版本和最终稿都经过校对和一致性检查
5. **自动合并有明确依据** — 以卖报小郎君为骨架，海明威补风景，大仲马补战斗，火星引力做局部强化，并且按写作单元逐个执行
6. **先看报告，再写正文** — `opus-report` 是放行条件和约束来源
7. **结构问题不上正文阶段硬修** — 需要新增/删除/重排关键剧情段时，回到 `novel-plan --revise`
8. **冗余描述在正文阶段清理** — 不把“水字数”主要寄希望于上游试写，正式成稿后必须再删一轮
