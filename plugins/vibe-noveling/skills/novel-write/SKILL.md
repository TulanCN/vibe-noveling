# 章节创作

## 何时使用

- 已有可写大纲，准备正式落正文
- 需要续写、改写或重写章节文本，但不想重做结构规划

## 调用边界

- 负责正文创作和文本层质检，结构级问题回退到 `novel-plan`
- 默认读取双层大纲和 Opus 报告，不直接替代前序规划流程
- 内置 27 个 AI 味检测项和最终稿文风矫正
- 优先读取 `可写场景纲要`
- 读取 `剧情思路卡` 作为结构约束
- 若只有旧版 `第三人称精简剧情纲要`，则降级兼容

## 目标

根据 `novel-plan` 生成的大纲（优先使用 `可写场景纲要` + `剧情思路卡`，兼容旧版 `第三人称精简剧情纲要`），并结合上游 Opus 测试报告提炼出的写作约束，默认创作全部 5 种写作风格的正文版本。5 个版本不是一次性并发派发，而是按固定顺序逐个调用 writer subagent，避免同一时刻同时发出 5 个请求。正式写作前，必须在正文阶段固定细切为 20 个剧情点。5 个中间稿都按同一组剧情点标题写作。最终稿合并也不是整章一次性融合，更不是句群拼贴，而是按 20 个剧情点顺序逐点择优，从 5 个风格版本中为每个剧情点选择 1 个主来源版本，再输出为显性保留剧情点标题并标记来源的 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`，用户确认满意后进入同步流程。

**核心特性**：

- 默认创作全部 5 种写作风格
- 按风格逐个调用 writer subagent，避免一次性派发 5 个请求
- 在正文阶段固定细切为 20 个剧情点
- 所有中间稿和最终稿都必须显性保留剧情点标题
- 按 20 个剧情点顺序逐点择优合并最终稿
- 最终稿每个剧情点都标记来源风格
- 自动合并生成最终稿
- 保留中间稿供回看和局部抽取
- 中间稿和最终稿都输出为带剧情点标题的章节正文

## 核心理念

章节写作负责三件事：

1. **把大纲变成血肉** — 在章节骨架上补环境、感官、动作链和台词
2. **把执行切分变成可核对骨架** — 先在正文阶段固定细切为 20 个剧情点；20 个剧情点不回写 outline，但要显性保留在所有稿件里，方便人工核对
3. **提供稳定的逐点择优** — 同一章先生成多种笔法，再按剧情点逐个选择最合适的版本合并成最终稿

上游大纲已经决定了：

- 本章职责
- 起始状态
- 结束目标
- 核心推进链
- 收尾点

正文写作负责：

- 把这些变成按剧情点分段、便于人工核对的章节正文
- 根据角色设定文件创作差异化台词
- 在不改大纲骨架的前提下，自己解决冗余、节奏和落文问题

## 最终稿偏好

最终稿不是五种风格的平均值。

最终稿应更接近当前项目已经人工调整过的正文气质，而不是追求五稿平均、四平八稳的合并腔。

优先保留主线压强、人物活气和网文阅读推进力：

- 优先保留主角锋芒
- 优先保留主角被区别对待
- 倒计时、任务压力、危险逼近、材料线索和主动决策优先保留
- 能让人物更鲜活的口语句、短反应、短态度，优先于平整但发干的概述句
- 能形成打脸效果、身份反差和态度翻转的段落优先保留
- 解释可以更直给一点，但只在它承担剧情信息、行动判断、战术理解或关系推进时保留

允许的提气方式：

- 允许角色即时反应、短促吐槽和带火气的接话
- 允许少量俏皮叙述和轻度作者式调侃
- 允许保留承担推进功能的直给解释

但必须守住边界：

- 吐槽要贴着人物、情境和压力来，不能像段子单独悬空
- 调侃要服务可读性和情绪温度，不能跳出当前 POV 炫机灵
- 默认压缩与主角装逼链条无关的安全句
- 不要把最终稿抹平成一个过分安全、过分中性的合并腔

## 内置 AI 味检测

`novel-write` 直接负责 27 个 AI 味检测项。主 skill 只保留执行时机和边界，完整清单和最终稿专属矫正规则统一放在 `references/ai-smell-checklist.md`。

执行原则：

- 中间稿写作时先按清单做轻量自检，但不提前统一抹平 5 个中间稿的风格差异
- 最终稿合并后先做一致性校验，再按完整清单逐项复检，并强制执行最终稿文风矫正
- 新增的检测项 22-27 吸收通用 humanizer 规则，不额外拆出独立 `/humanizer` skill
- 只吸收适合中文网文正文的通用 AI 腔规则，不机械照搬英文排版约束
- 破折号限制按整章统计，只统计中文破折号 `——`
- 单章最多保留 1 处破折号；超过时必须改写句式，用断句、动作或对白节奏替代，不能只删除标点了事
- `但是`、`然而`、`不过`、`只是`、`可`、`却`、`然后`、`随后`、`紧接着`、`两秒后` 等过渡性词语或短语要统一复检；凡删除后不影响上下文衔接、语义转折、时间感和阅读理解的，直接删除
- 合并后正文复检时，要先评估当前章节里所有角色的细节描写和行为描写；如果删除后对主角的剧情推进、压力变化、关系判断和线索释放无任何影响，直接删除
- 相邻短句若语义连贯、主语相同或动作连续，优先合并并重写；保留戏剧性节拍和独立短句留白，不做机械并句
- 合并短句时要做因果重写：解释性短句优先改成先因后果；主语一致的连续动作要改成更有画面感的表达，而不是简单用逗号串联
- 物品描写不得先写物品名称，再用短句补解释特征；要直接描写它是什么样的东西，而不是先说是什么、再说特征
- AI 味检测通过后，必须再执行一轮独立润色；只升级词汇质感和动作表现，不改事实信息、动作因果和剧情点职责
- 润色阶段要统一替换物品描写、环境描写、人物描写中的普通词汇，改成更优美、精美、华丽、有高级感的表达
- 润色阶段要统一替换人物行为描写中的普通动词，改成更豪迈、精准、刺激、带武侠风味的表达
- 水字数检查在正文阶段只处理句子、段落、场景内部铺陈级冗余，不负责删除整个剧情点或重排结构
- 如果检测结果显示必须改动结构层推进，停止正文硬修，回到 `novel-plan --revise`

```bash

```

## 完整流程

```text
  第一步：获取上下文（缓存优先）
       │  检查章节目录中的 上下文.md 是否存在且有效
       │  ├─ 缓存有效 → 直接复用
       │  └─ 缓存无效 → 调用 context-collector
       ▼
  第二步：读取大纲
       │  优先读取 可写场景纲要 + 剧情思路卡
       │  若仅有旧版大纲，则降级读取第三人称精简剧情纲要
       ▼
  第二步 A：正文阶段细化剧情点
       │  优先从可写场景纲要细切出恰好 20 个剧情点
       │  旧版大纲兼容时从精简剧情纲要细切
       │  20 个剧情点不回写 outline，但必须显性保留在全部稿件中
       ▼
  第二步 B：读取 Opus 测试报告（如存在）
       │  提炼“写作约束摘要”
       ▼
  第三步：默认全风格写作准备
       │  默认创作全部 5 种写作风格
       ▼
  第四步：逐个调用 writer subagent + 校对
       │  按固定顺序逐个调用 writer subagent
       │  不要一次性同时派发 5 个请求
       │  每个 writer subagent：写完整章节 → 轻量质检 → 冗余清理 → 修改（最多 3 轮）
       ▼
  第五步：中间稿一致性检查
       │  对每个版本调用 consistency-guard
       ▼
  第六步：按剧情点逐点择优合并最终稿
       │  按 20 个剧情点顺序逐点择优合并
       │  每次只处理当前剧情点在 5 个风格版本中的对应内容
       │  每个剧情点只选 1 个主来源版本，并在标题中标记来源风格
       ▼
  第七步：合并后最终稿一致性校验
       │  对 chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md 先调用 consistency-guard
       │  先修补称谓、时间线、认知和设定冲突，再进入正文质检
       ▼
  第八步：合并后正文 AI 味检测与文风复检
       │  对 chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md 再做一轮 27 项 AI 味检测
       │  执行一次定点修正 AI 味的正文后矫正
       ▼
  第九步：AI 味清理后的正文润色
       │  基于去除 AI 味后的正文做一轮词汇升级和动作润色
       │  描写词汇高级化，行为词汇武侠化，但不改事实与因果
       ▼
  第十步：交付最终稿和中间稿
       │  输出最终稿路径和中间稿路径
       ▼
  第十一步：用户确认后触发同步
       │  用户通知“确认 chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md”
       │  自动执行 novel-sync
       ▼
  ✅ 完成
```

## 第一步：获取上下文（优先使用缓存）

### 1.1 检查上下文缓存

首先检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md` 是否存在且有效。

缓存有效性判断：

- 文件存在 → 读取缓存头的 `生成时间`
- 检查相关设定文件的修改时间是否晚于缓存生成时间：
  - 出场角色的设定文件（`memory/entities/characters/`）
  - `memory/worldbuilding.md`
  - `memory/_graph.json`
- 所有文件都未修改 → **缓存有效**
- 有文件被修改 → **缓存失效，重新收集**

### 1.2 根据缓存状态选择路径

**缓存有效**：

```text
📖 使用上下文缓存

检测到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md（缓存有效）
生成时间：{timestamp}
设定文件无变更 → 直接复用
```

**缓存不存在或已失效**：

```text
📖 缓存不可用

{缓存不存在 / 缓存已失效：角色设定文件已变更}

调用 context-collector 重新收集上下文...
```

调用方式：

```text
Agent 工具，subagent_type: context-collector
prompt: 为第 X 章收集上下文，大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md
```

### 1.3 设定缺失处理

如果 context-collector 报告设定缺失，必须先补全设定，再继续写作。

禁止：

- 忽略设定缺失
- 自己编造设定
- 假设“后续再补”

## 第二步：读取大纲

从 `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` 读取本章大纲。

**大纲关键认知**：

- 上游优先给的是 `剧情思路卡` + `可写场景纲要`
- `剧情思路卡` 用来约束本章核心兑现、冲突链和信息释放边界
- `可写场景纲要` 用来承接触发事件、冲突落地、转折点与落点变化
- 若章节尚未升级，上游给的是**第三人称精简剧情纲要**，按兼容路径降级读取
- 大纲的目标是让作者易读、易改，不是给正文执行用的分镜表
- 你要从中提取推进链，而不是要求上游已经把它切成剧情点

**重点读取内容**：

- 基本信息
- 起始状态
- 剧情思路卡
- 可写场景纲要
- 结束目标
- 章节任务卡
- 第三人称精简剧情纲要（仅旧版兼容路径必读）
- 伏笔处理

**禁止**：

- 擅自改变章节收尾目标
- 擅自改掉关键推进链
- 忽略 `剧情思路卡` 的本章边界与核心兑现约束
- 把大纲重新解释成另一个不同的大纲

## 第二步 A：正文阶段细化剧情点

在正式写作前，先在正文阶段固定细切为 20 个剧情点。

### 2A.1 切分目标

优先从 `可写场景纲要` 细切出恰好 20 个**剧情点**；若仅存在旧版结构，则从 `第三人称精简剧情纲要` 细切。每个剧情点只需要回答五件事：

1. 当前剧情点的标题是什么
2. 这一点从什么动作或状态开始
3. 这一点的核心冲突或信息推进是什么
4. 这一点的关系变化或情绪压力是什么
5. 这一点把压力推到哪里结束

### 2A.2 剧情点规则

- 剧情点数量固定为 20，不允许多于或少于 20 个
- 每个剧情点都必须承担一次明确的动作推进、信息释放、关系位移或压力变化
- 单个剧情点可以短，但不能空转，不能拿一串连续小动作硬凑成无意义切片
- 20 个剧情点不回写 outline
- 所有中间稿和最终稿都必须显性保留剧情点标题
- 中间稿标题格式统一为 `【剧情点01：标题】`
- 最终稿标题格式统一为 `【剧情点01：标题｜来源：海明威】`
- 同一个剧情点内部可以自由扩写台词、动作和细节
- 如果切分后发现必须新增关键事件、删除关键事件或重排章节收尾，停止并回到 `novel-plan --revise`

### 2A.3 何时必须退回 `novel-plan`

以下情况必须停止正文阶段硬修：

- 需要新增关键剧情段
- 需要删除整段核心推进
- 需要重排关键剧情顺序
- 需要改动章节收尾目标
- 需要把伏笔处理位置整体后挪或前移

## 第二步 B：读取 Opus 测试报告（如存在）

如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`，必须在正式写正文前先读取它。

这份报告的作用有两个：

1. **放行判定**：这一章现在能不能直接进入正文创作
2. **写作约束**：正式写作时必须补出、必须避免、必须压缩的点

### 2B.1 放行规则

**如果不存在报告**：

```text
📎 未找到 Opus 测试报告

本次按 outline + context 直接写作。
风险：不会自动继承上游试写发现的问题点。
```

**如果报告判定为 `需先重做大纲`**：

立即停止，不进入风格写作阶段，直接回到 `/novel-plan --revise`。

**如果报告判定为 `需小修后写`**：

只有当这些问题能被转成**局部写作约束**时，才允许继续写。否则回到 `novel-plan --revise`。

### 2B.2 约束提炼方式

传给写作 agent 的应该是短的“写作约束摘要”，而不是整份诊断报告。摘要应包含：

- 必须补出的桥接动作
- 必须压缩的信息段
- 需要加固的角色声线
- 禁止重犯的问题

## 第三步：默认全风格写作准备

默认创作以下 5 种风格版本：

1. **海明威风（极简硬派）**
2. **卖报小郎君风（轻悬疑群像）**
3. **会说话的肘子风（现代快节奏战斗 + 临场决断）**
4. **大仲马风（精密布局与戏剧交锋）**
5. **吐槽口语风（主角反应 + 口语提气）**

中间稿全部保留到本章目录 `chapters/vol-{volume_padded}/ch-{chapter_padded}/`，供回看、局部抽取和后续复用。

## 第四步：逐个调用 writer subagent + 校对

按以下固定顺序逐个调用 writer subagent：

1. 海明威 → `writer-hemingway`
2. 卖报小郎君 → `writer-maibaoxiaolangjun`
3. 会说话的肘子 → `writer-zhouzi`
4. 大仲马 → `writer-dazhongma`
5. 吐槽口语风 → `writer-banter`

启动规则：

- 每次只调用一个新的 writer subagent
- 前一个 writer subagent 成功接收并完成任务后，再调用下一个
- 不要一次性同时派发 5 个请求，避免请求失败
- 5 个风格版本都完成基础校对后，再进入后续一致性检查和自动合并

调用时使用 Agent 工具，并将对应风格路由到固定 subagent_type：

```text
海明威 → `subagent_type: writer-hemingway`
卖报小郎君 → `subagent_type: writer-maibaoxiaolangjun`
会说话的肘子 → `subagent_type: writer-zhouzi`
大仲马 → `subagent_type: writer-dazhongma`
吐槽口语风 → `subagent_type: writer-banter`
```

writer subagent 只负责稳定文风人格和风格自检；本章的动态输入、输出路径、剧情点摘要、上下文摘要和 Opus 约束仍由 `novel-write` 主 skill 统一组装并下发。

### 动态写作任务 Prompt 模板

```text
你是一个网文写作 agent。请按以下要求创作第 {chapter} 章的正文。

## 文件写入要求

你必须将最终完成的正文写入：
`chapters/vol-{volume_padded}/ch-{chapter_padded}/{style_name}.md`

## 基本信息
- 章节：第 {chapter} 章
- 写作风格：{style_name}
```

## 第六步：按剧情点逐点择优合并最终稿

自动合并不是把整章一次性糅成一个版本，也不是从多个版本里抽句拼贴，而是按 20 个剧情点顺序逐点择优合并。

合并执行方式：

- 按 20 个剧情点顺序逐点择优合并
- 每次只处理当前剧情点在 5 个风格版本中的对应内容
- 每个剧情点只选 1 个主来源版本
- 只允许对选中的剧情点做必要的局部修字，不从其他版本拼句
- 最终稿必须在剧情点标题中标记来源风格

自动合并必须遵循以下固定规则：

1. **线索推进、群像配合、章末钩子优先卖报小郎君风**：当前剧情点需要线索组织和阅读推进力时，优先选择卖报小郎君版本
2. **环境克制、动作简洁、少修辞优先海明威风**：当前剧情点需要冷硬质感和直写效果时，优先选择海明威版本
3. **正面冲击、速度感、临场决断优先会说话的肘子风**：当前剧情点需要现代快节奏战斗、危险逼近和干净利落的判断反馈时，优先选择会说话的肘子版本
4. **布局交锋、信息差、战斗推演优先大仲马风**：当前剧情点需要戏剧交锋和攻防层次时，优先选择大仲马版本
5. **即时反应、短吐槽、口语提气优先吐槽口语风**：当前剧情点需要人物活气、主角反应和短促接话时，优先选择吐槽口语风版本
6. **如果角色活气不够、信息已经对了但读起来发平、缺少即时反应，就自动从吐槽口语风中抽取当前剧情点作为优先候选来源**
7. **不能把紧张场面卸力成段子腔**：吐槽口语风只能提气，不能为了热闹把危险、悬念和冲突写轻
8. **每个剧情点只选 1 个主来源版本**：不要把多个版本句群拼接成一个平均化段落
9. **若 5 个版本都不理想，只选最接近目标的一个做局部修字**：允许定点改句，但不允许重新变回跨版本拼贴
10. **择优时优先选择更接近当前项目已经人工调整过的正文气质的版本**：不要为了整齐把人物活气和网文推进力洗掉
11. **优先保留主角锋芒、主角被区别对待和打脸效果**：如果几个版本信息量接近，优先选择更能放大主角特殊性和压制反转的版本
12. **默认压缩与主角装逼链条无关的安全句**：不为了平稳而保留那些既不增强主角锋芒、也不推进兑现的概述句
13. **必须在最终稿剧情点标题中标记来源风格**：例如 `【剧情点07：雨巷试探｜来源：大仲马】`
14. **允许保留承担推进功能的直给解释**：若解释承担剧情信息、行动决策、战术理解或关系变化，不要机械删掉
15. **所有剧情点合并后仍要保持 20 个标题完整可核对**：不能在最终稿里再把它们抹平成无标题正文
