# 章节创作

## 何时使用

- 已有可写大纲，准备正式落正文
- 需要续写、改写或重写章节文本，但不想重做结构规划

## 调用边界

- 负责正文创作和文本层质检，结构级问题回退到 `novel-plan`
- 默认读取双层大纲和 Opus 报告，不直接替代前序规划流程
- 内置 10 个 AI 味检测项和最终稿文风矫正
- `CLAUDE.md` 中的"小说宪法"是写作阶段的最高准则；宪法中的"作者人格"维度（叙事姿态、情感温度、幽默方式、信息习惯、叙事癖好）直接影响正文合并时的语气收束方向；宪法优先于默认规则和风格优先规则
- 优先读取 `可写场景纲要`
- 读取 `剧情思路卡` 作为结构约束
- 若只有旧版 `第三人称精简剧情纲要`，则降级兼容

## 目标

根据 `novel-plan` 生成的大纲（优先使用 `可写场景纲要` + `剧情思路卡`，兼容旧版 `第三人称精简剧情纲要`），并结合上游 Opus 测试报告提炼出的写作约束，默认创作 2 种写作风格的正文版本。2 个版本不是一次性并发派发，而是按固定顺序逐个调用 writer subagent，避免同一时刻同时发出多个请求。正式写作前，必须在正文阶段固定细切为 20 个剧情点。2 个中间稿都按同一组剧情点标题写作。最终稿合并也不是整章一次性融合，更不是句群拼贴，而是按 20 个剧情点顺序逐点择优，默认选择肘子版本，高光剧情点优先选择大仲马版本；合并规则支持章节级配置（通过 `合并配置.md`），无配置时使用 `references/merge-rules.md` 中的默认规则。最终稿输出为显性保留剧情点标题并标记来源的 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`，用户确认满意后进入同步流程。

**核心特性**：

- 默认创作 2 种写作风格（肘子底稿 + 大仲马高光）
- 按风格逐个调用 writer subagent，避免一次性派发多个请求
- 在正文阶段固定细切为 20 个剧情点
- 所有中间稿和最终稿都必须显性保留剧情点标题
- 按剧情点逐点择优合并最终稿，合并规则支持章节级配置
- 最终稿每个剧情点都标记来源风格
- 自动合并生成最终稿
- 保留中间稿供回看和局部抽取
- 中间稿和最终稿都输出为带剧情点标题的章节正文
- 交付后提示用户运行 `/novel-revise` 进行返修，返修完成后运行 `/novel-sync` 进行同步

## 核心理念

章节写作负责三件事：

1. **把大纲变成血肉** — 在章节骨架上补环境、感官、动作链和台词
2. **把执行切分变成可核对骨架** — 先在正文阶段固定细切为 20 个剧情点；20 个剧情点不回写 outline，但要显性保留在所有稿件里，方便人工核对
3. **提供稳定的逐点择优** — 同一章先生成多种笔法，再按剧情点逐个选择最合适的版本合并成最终稿

上游大纲已经决定了：

- 本章职责
- 起始状态
- 结束目标
- 核心推进链
- 收尾点

正文写作负责：

- 把这些变成按剧情点分段、便于人工核对的章节正文
- 根据角色设定文件创作差异化台词
- 在不改大纲骨架的前提下，自己解决冗余、节奏和落文问题

## 最终稿偏好

最终稿偏好规则统一存放在 `references/merge-rules.md` 的"最终稿偏好"段落中。本处不内联展开，合并时从该文件读取。

## 内置 AI 味检测

AI 味检测由主 session 直接执行，读取 `references/ai-smell-checklist.md` 作为唯一检测清单来源。完整检测清单和矫正规则从该文件读取，不内联展开。

执行原则：

- 中间稿写作时先做轻量自检，但不提前统一抹平 2 个中间稿的风格差异
- 最终稿合并后先做一致性校验，再读取 `references/ai-smell-checklist.md` 执行 28 项完整检测与定点修正
- 主 session 按清单逐项检测、定点修正、写回正文
- AI 味检测只处理文本层问题；如果发现结构性问题，停止正文硬修，回到 `novel-plan --revise`
- AI 味检测通过后，执行一轮独立润色；只升级词汇质感和动作表现，不改事实信息、动作因果和剧情点职责
- 润色阶段要统一替换物品描写、环境描写、人物描写中的普通词汇，改成更优美、精美、华丽、有高级感的表达
- 润色阶段要统一替换人物行为描写中的普通动词，改成更豪迈、精准、刺激、带武侠风味的表达
- 润色完成后，执行仿人工增强：统计正文全部 7 类虚词密度，与手工精修基准对比，对低于基准的虚词类别用 SSoT 做定向补插（允许改写句式），使正文虚词分布接近手工精修水平

## 使用方式

```bash
/novel-write 01
/novel-write 02
/novel-write next
```

## 完整流程

```text
  第一步：获取上下文（缓存优先）
       │  检查章节目录中的 上下文.md 是否存在且有效
       │  ├─ 缓存有效 → 直接复用
       │  └─ 缓存无效 → 调用 context-collector
       ▼
  第二步：读取大纲
       │  优先读取 可写场景纲要 + 剧情思路卡
       │  若仅有旧版大纲，则降级读取第三人称精简剧情纲要
       ▼
  第二步 A：正文阶段细化剧情点
       │  优先从可写场景纲要细切出恰好 20 个剧情点
       │  旧版大纲兼容时从精简剧情纲要细切
       │  20 个剧情点不回写 outline，但必须显性保留在全部稿件中
       ▼
  第二步 B：读取 Opus 测试报告（如存在）
       │  提炼”写作约束摘要”
       ▼
  第二步 C：合并配置
       │  检查 合并配置.md 是否存在
       │  ├─ 已存在 → 直接复用
       │  └─ 不存在 → 询问用户合并偏好 → 生成 合并配置.md
       ▼
  第三步：双风格写作准备
       │  默认创作 2 种写作风格（肘子底稿 + 大仲马高光）
       │  询问用户并发数（默认 1 并发）
       ▼
  第四步：调用 writer subagent + 校对
       │  按固定顺序调用 writer subagent
       │  根据用户选择的并发数分批派发
       │  每个 writer subagent：写完整章节 → 轻量质检 → 冗余清理 → 修改（最多 3 轮）
       ▼
  第五步：中间稿一致性检查
       │  对每个版本调用 consistency-guard
       ▼
  第六步：按剧情点逐点择优合并最终稿
       │  按 20 个剧情点顺序逐点择优合并
       │  每次只处理当前剧情点在 2 个风格版本中的对应内容
       │  每个剧情点只选 1 个主来源版本，并在标题中标记来源风格
       ▼
  第七步：合并后最终稿一致性校验
       │  对 chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md 先调用 consistency-guard
       │  先修补称谓、时间线、认知和设定冲突，再进入正文质检
       ▼
  第八步：合并后正文 AI 味检测与文风复检
       │  读取 references/ai-smell-checklist.md 执行 28 项检测与定点修正
       │  主 session 逐项检测、定点修正、写回正文
       ▼
  第九步：AI 味清理后的正文润色
       │  基于去除 AI 味后的正文做一轮词汇升级和动作润色
       │  描写词汇高级化，行为词汇武侠化，但不改事实与因果
       ▼
  第十步：仿人工增强（统计驱动 + SSoT 定向补偿）
       │  统计正文虚词密度 → 与手工精修基准对比 → 对缺口类别做定向补插
       │  简单虚词直接插入，介词/副词/判断动词/连词允许改写句式
       │  SSoT 驱动随机选择位置，确保自然分散
       │  改动后写回正文
       ▼
  第十一步：交付最终稿和中间稿
       │  输出最终稿路径和中间稿路径
       │  提示用户运行 /novel-revise 进行返修
       ▼
  ✅ 完成（等待用户返修或同步）
```

## 第一步：获取上下文（优先使用缓存）

### 1.1 检查上下文缓存

首先检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md` 是否存在且有效。

缓存有效性判断：

- 文件存在 → 读取缓存头的 `生成时间`
- 检查相关设定文件的修改时间是否晚于缓存生成时间：
  - 出场角色的设定文件（`memory/entities/characters/`）
  - `memory/worldbuilding.md`
  - `memory/_graph.json`
- 所有文件都未修改 → **缓存有效**
- 有文件被修改 → **缓存失效，重新收集**

### 1.2 根据缓存状态选择路径

**缓存有效**：

```text
📖 使用上下文缓存

检测到 chapters/vol-{volume_padded}/ch-{chapter_padded}/上下文.md（缓存有效）
生成时间：{timestamp}
设定文件无变更 → 直接复用
```

**缓存不存在或已失效**：

```text
📖 缓存不可用

{缓存不存在 / 缓存已失效：角色设定文件已变更}

调用 context-collector 重新收集上下文...
```

调用方式：

```text
Agent 工具，subagent_type: context-collector
prompt: 为第 X 章收集上下文，大纲文件：chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md
```

### 1.3 设定缺失处理

如果 context-collector 报告设定缺失，必须先补全设定，再继续写作。

禁止：

- 忽略设定缺失
- 自己编造设定
- 假设“后续再补”

## 第二步：读取大纲

从 `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md` 读取本章大纲。

**大纲关键认知**：

- 上游优先给的是 `剧情思路卡` + `可写场景纲要`
- `剧情思路卡` 用来约束本章核心兑现、冲突链和信息释放边界
- `可写场景纲要` 用来承接触发事件、冲突落地、转折点与落点变化
- 若章节尚未升级，上游给的是**第三人称精简剧情纲要**，按兼容路径降级读取
- 大纲的目标是让作者易读、易改，不是给正文执行用的分镜表
- 你要从中提取推进链，而不是要求上游已经把它切成剧情点

**重点读取内容**：

- 基本信息
- 起始状态
- 剧情思路卡
- 可写场景纲要
- 结束目标
- 章节任务卡
- 第三人称精简剧情纲要（仅旧版兼容路径必读）
- 伏笔处理

**禁止**：

- 擅自改变章节收尾目标
- 擅自改掉关键推进链
- 忽略 `剧情思路卡` 的本章边界与核心兑现约束
- 把大纲重新解释成另一个不同的大纲

## 第二步 A：正文阶段细化剧情点

在正式写作前，先在正文阶段固定细切为 20 个剧情点。

### 2A.1 切分目标

优先从 `可写场景纲要` 细切出恰好 20 个**剧情点**；若仅存在旧版结构，则从 `第三人称精简剧情纲要` 细切。每个剧情点只需要回答五件事：

1. 当前剧情点的标题是什么
2. 这一点从什么动作或状态开始
3. 这一点的核心冲突或信息推进是什么
4. 这一点的关系变化或情绪压力是什么
5. 这一点把压力推到哪里结束

### 2A.2 剧情点规则

- 剧情点数量固定为 20，不允许多于或少于 20 个
- 每个剧情点都必须承担一次明确的动作推进、信息释放、关系位移或压力变化
- 单个剧情点可以短，但不能空转，不能拿一串连续小动作硬凑成无意义切片
- 20 个剧情点不回写 outline
- 所有中间稿和最终稿都必须显性保留剧情点标题
- 中间稿标题格式统一为 `【剧情点01：标题】`
- 最终稿标题格式统一为 `【剧情点01：标题｜来源：肘子】`
- 同一个剧情点内部可以自由扩写台词、动作和细节
- 如果切分后发现必须新增关键事件、删除关键事件或重排章节收尾，停止并回到 `novel-plan --revise`

### 2A.3 密度继承与传导

剧情点从所属场景块继承密度标记（高/中/低），并传导至写作约束。

**继承规则**：

- 从大纲的可写场景纲要中读取每个场景块的密度字段
- 每个剧情点继承其所属场景的密度
- 如果场景未标密度，继承基本信息中的本章密度定位
- 如果本章密度也未标注，默认为中（标准）

**密度对剧情点的影响**：

- **高密度剧情点**：字数空间更大（200-400字/点），必须包含感官层次（视觉+听觉+触觉）、在场角色反应、主角内心冲击
- **中密度剧情点**：标准处理（120-200字/点）
- **低密度剧情点**：压缩处理（50-100字/点），只保留动作推进和信息释放，禁止环境描写、配角心理、微动作、过场废话

**传导方式**：

密度标记写入 `剧情点.md`，每个剧情点条目追加密度字段，格式如下：

```markdown
### 剧情点 03：[标题]
- 密度：高（展开）← 继承自场景 2
- 起始状态：
- 核心推进：
- 关系变化：
- 结束压力：
```

### 2A.3 何时必须退回 `novel-plan`

以下情况必须停止正文阶段硬修：

- 需要新增关键剧情段
- 需要删除整段核心推进
- 需要重排关键剧情顺序
- 需要改动章节收尾目标
- 需要把伏笔处理位置整体后挪或前移

### 2A.4 剧情点落盘

将细切完成的 20 个剧情点写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/剧情点.md`。

格式要求：
- 每个剧情点包含：编号、标题、起始状态、核心推进、关系变化、结束压力
- 此文件是所有 writer subagent 和合并 agent 的共享输入
- 落盘后不再通过 prompt 内联传递

## 第二步 B：读取 Opus 测试报告（如存在）

如果存在 `chapters/vol-{volume_padded}/ch-{chapter_padded}/Opus报告.md`，必须在正式写正文前先读取它。

这份报告的作用有两个：

1. **放行判定**：这一章现在能不能直接进入正文创作
2. **写作约束**：正式写作时必须补出、必须避免、必须压缩的点

### 2B.1 放行规则

**如果不存在报告**：

```text
📎 未找到 Opus 测试报告

本次按 outline + context 直接写作。
风险：不会自动继承上游试写发现的问题点。
```

**如果报告判定为 `需先重做大纲`**：

立即停止，不进入风格写作阶段，直接回到 `/novel-plan --revise`。

**如果报告判定为 `需小修后写`**：

只有当这些问题能被转成**局部写作约束**时，才允许继续写。否则回到 `novel-plan --revise`。

### 2B.2 约束提炼方式

传给写作 agent 的应该是短的“写作约束摘要”，而不是整份诊断报告。摘要应包含：

- 必须补出的桥接动作
- 必须压缩的信息段
- 需要加固的角色声线
- 禁止重犯的问题

### 2B.3 约束落盘

将提炼的"写作约束摘要"写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/写作约束.md`。
若不存在 Opus 报告，写入"无额外约束"。

密度约束同时追加到 `写作约束.md`，格式如下：

```markdown
## 详略约束

高密度剧情点（{编号列表}）：
  - 字数空间：200-400字/点
  - 必须包含：感官层次、在场角色反应、主角内心冲击
  - 禁止压缩，宁可写多

低密度剧情点（{编号列表}）：
  - 字数空间：50-100字/点
  - 只保留：动作推进 + 信息释放
  - 禁止：环境描写、配角心理、微动作、过场废话

中密度剧情点（{编号列表}）：
  - 字数空间：120-200字/点
  - 标准处理
```

## 第二步 C：合并配置

在剧情点细切完成后、正式写作前，确认本章的合并配置。

### 2C.1 检查已有配置

检查 `chapters/vol-{volume_padded}/ch-{chapter_padded}/合并配置.md` 是否存在。

**如果已存在**：读取并确认配置内容，进入第三步。

### 2C.2 协助创建配置（无已有配置时）

如果 `合并配置.md` 不存在，**必须使用 `AskUserQuestion` 工具**向用户询问本章的合并偏好。

调用示例：

```text
AskUserQuestion({
  questions: [{
    header: "合并配置",
    multiSelect: false,
    options: [
      { label: "采纳建议", description: "使用基于本章大纲自动生成的合并偏好建议" },
      { label: "手动调整", description: "我想自己指定风格映射和基调偏好" },
      { label: "使用默认规则", description: "跳过配置，完全使用 references/merge-rules.md 默认规则" }
```

### 8.1 检测重点关注

- **段落级节奏**：一个段落内出现 3 个及以上主语一致的连续短句，必须重新改写为一句通顺流畅的长句
- **检测项 1**：情感与心理描写 — 用动作、感官和结果替代情感标签和大段心理解释
- **检测项 2**：对话失真 — 对话要有刺和冲突，不要礼貌协商、解释设定或全员理性
- **检测项 3**：空洞修饰 — 砍掉万能形容词、修辞堆砌、过量明喻和不推进的平淡比喻
- **检测项 4**：结构与节奏 — 关键场面放大、过渡压缩、结尾留悬念、砍空转桥段和强行三连
- **检测项 5**：无功能细节 — 声音描写、微动作（嘴角手指等）不承担叙事功能就删
- **检测项 6**：AI 腔句式与警示词 — 高频过渡词、破折号滥用、重复率、数字硬撑、否定排比
- **检测项 7**：水字数检查 — 以段落为单元扫描，删除后不影响主角推进的直接删除；段落内短句碎片优先合并
- **检测项 8**：虚假拔高 — 意义拔高、宣传腔、伪深刻分析、空泛权威，一律改回动作和后果
- **检测项 9**：交流残留 — 对话助手口吻、说明书口吻、服务腔不能混进正文
- **检测项 10**：翻译腔 — 英语语法结构照搬中文的各类问题

### 8.2 特殊处理规则

**破折号限制**：

- 按整章统计，只统计中文破折号 `——`
- 单章最多保留 1 处破折号
- 超过时必须改写句式，用断句、动作或对白节奏替代，不能只删除标点了事

**过渡词处理**：

- `但是`、`然而`、`不过`、`只是`、`可`、`却`、`然后`、`随后`、`紧接着`、`两秒后` 等过渡性词语统一当作可疑项
- 删除后不影响上下文衔接、语义转折、时间感和阅读理解的，直接删除
- 确实承担转折功能的，检查能否改成动作递进、结果反差、信息落差或自然断句

**水字数处理边界**：

- 只处理句子、段落、场景内部铺陈级冗余
- 如果发现必须删除整个剧情点或重排结构，**停止并回到 `novel-plan --revise`**
- 相邻短句合并时要做因果重写，不是简单用逗号串联

**保留白名单**（不因”不够文”就删除）：

- 犀利吐槽
- 贴着场景的轻俏叙述
- 承担剧情信息的直给解释
- 带人物态度的口语短句

保留前提：必须贴着当前 POV、当前动作和当前压力。

**清理黑名单**（即使热闹也清掉）：

- 连续玩梗
- 脱离视角的作者炫机灵
- 硬插的段子句
- 动作已说明白还再解释的空转说明
- 把危险/悬念/冲突卸力成轻飘调侃

### 8.3 检测报告

## 第九步：AI 味清理后的正文润色

在 AI 味检测与文风复检完成后，再对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 执行一轮独立润色。

执行要求：

1. 物品描写：避免只写“普通、简单、破旧、发亮”这类直白词，优先替换成更有质感、更有层次的华丽表达
   物品描写不要先报物品名称，再用短句补特征；要直接把它写成“是什么样的东西”
2. 环境描写：优先补出氛围、光泽、纹理、层次和压迫感，让场景显得更精美、更高级
3. 人物描写：优先使用更有风骨、气势、锋芒和仪态感的词汇，不把人物描写写得发平
4. 行为描写：把角色的动作描写，改写成带武侠风味的动作表达
5. 润色后的句子仍必须服务主角推进，不得因为追求华丽词汇而让动作发虚、信息发糊

## 第十步：仿人工增强（统计驱动 + SSoT 定向补偿）

在润色完成后、交付前，对 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md` 执行一轮仿人工增强。

### 执行方式

本步由主 session 直接执行，不委托 agent。

#### 第一阶段：虚词密度统计

1. 读取正文，统计全文总字数（不含标点和空格）
2. 统计以下全部 7 类虚词的出现次数：
   - **动态助词**：了、着、过
   - **结构助词**：的、地、得
   - **语气词**：么、吧、呢、啊、嘛、哦、哈、呀
   - **介词**：在、从、向、把、被、对、为、给、用、比、按
   - **副词**：就、还、都、也、又、才、再、更、最、很、真、好、太、已经、正在、刚刚
   - **判断动词**：是、有
   - **连词**：而、但、却、且、或
3. 计算每类虚词的密度（次/100字）

#### 第二阶段：缺口计算

与手工精修基准对比：

| 虚词类别 | 基准密度(/100字) | 内部分配 |
|----------|------------------|----------|
| 动态助词 | 3.15 | 了:着:过 ≈ 60:26:14 |
| 结构助词 | 3.12 | 的:地:得 ≈ 68:27:4 |
| 介词 | 2.68 | 在/从/向/把/被 为主 |
| 副词 | 2.44 | 就/还/都/也 为主 |
| 判断动词 | 1.22 | 是/有 |
| 语气词 | 0.51 | 么/吧/呢 为主 |
| 连词 | 0.27 | 而/但/却 为主 |

对每个子类计算缺口：
- 缺口 = max(0, 基准密度 - 当前密度) × 总字数 / 100
- 缺口取整数，向上取整
- 某子类当前密度已达到或超过基准的，不做补插
- 额外操作：删句号（目标：约每 8-10 个句号删 1 个，用 SSoT 决定删哪些）

#### 第三阶段：SSoT 定向补偿

1. 在心里生成一个复杂随机字符串（至少 30 字符），不暴露给用户
2. 根据缺口数量，用随机字符串的不同片段做 rolling hash，为每个需要补插的虚词随机选择一个插入位置（句子序号）
3. 确保同一个句子最多只被选中一次（避免堆叠）
4. 在选中位置执行补偿：

   **简单插入类**（不改句式，直接在合适位置插入虚词）：
   - **加「了」**：在最近动词后或分句末插入，表示完成态或变化
   - **加「着」**：在最近动词后插入，表示持续态
   - **加「过」**：在最近动词后插入，表示经历态
   - **加「的」**：在名词短语后或分句末插入
   - **加「地」**：在动词前的状语位置插入（XX地做某事）
   - **加「得」**：在动词后的补语位置插入（做得XX）
   - **加语气词**：在句末标点前插入，增加口语感
   - **加副词**：在动词或形容词前的合适位置插入

   **句式改写类**（允许调整句式来自然补入虚词，但必须保持原意不变）：
   - **补介词**：改写句式引入介词结构（如「他走进房间」→「他从门外走进了房间」）
   - **补判断动词**：改写句式引入判断结构（如「他很厉害」→「他是个厉害角色」）
   - **补连词**：在分句间插入连词（如「他赢了，没人不服」→「他赢了，但没人不服」）

   **其他操作**：
   - **删句号**：删除句末「。」，让相邻句子自然流接（只删「。」，不删「！」「？」「……」）

#### 第四阶段：验证与写回

1. 重新统计虚词密度，确认各补插类别的密度已接近基准
2. 如果某个类别仍明显低于基准，再做一轮补充
3. 如果某个类别补插过度（超过基准 150%），回退多余的插入
4. 处理完成后写回 `正文.md`

### 跳过规则

以下内容不参与仿人工增强：
- 剧情点标题行（`【剧情点...】` 格式）
- 空行
- 纯分隔行（`***`）

### 执行边界

- 简单虚词做定点插入，不改句式
- 介词/判断动词/连词的补入允许改写句式，但必须保持原意不变、不改变事实信息和动作因果
- 不对同一段落内的句子做合并或拆分
- 如果某类虚词的缺口为 0，直接跳过该类别

## 第十一步：交付最终稿和中间稿

默认交付以下文件：

- 最终稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/会说话的肘子.md`
- 中间稿：`chapters/vol-{volume_padded}/ch-{chapter_padded}/大仲马.md`

交付说明中应简短说明：

- 20 个剧情点各自选用了哪个来源风格
- 哪些剧情点使用了大仲马版本（高光段落）
- 是否只做了局部修字，而没有跨风格拼贴

交付后提示用户：


## 字数统计

**字数要求**：

- 每个中间稿：2500-4000 字
- 最终合并稿：允许在 2500-4500 字之间，只要节奏更紧、不水字数即可

## 输出格式

每个中间稿和最终合并稿都必须输出为**带剧情点标题的章节正文**：

- 中间稿标题格式：`【剧情点01：标题】`
- 最终稿标题格式：`【剧情点01：标题｜来源：肘子】`
- 标题下使用自然段落
- 允许必要的场景换行或 `***` 作为弱分隔
- 不得隐藏剧情点标题，不得把 20 个剧情点抹平

### 文件写入时机

1. **准备阶段**：细切完成后将剧情点写入 `剧情点.md`，将写作约束写入 `写作约束.md`
2. **创作阶段**：每个 Agent 自检通过后立即将正文写入 `chapters/vol-{volume_padded}/ch-{chapter_padded}/{style_name}.md`
3. **合并阶段**：按 20 个剧情点顺序读取全部中间稿的对应内容，逐点择优生成最终稿 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`
4. **润色阶段**：AI 味检测与修正完成后，主 session 对最终稿执行一轮独立润色，再回写 `chapters/vol-{volume_padded}/ch-{chapter_padded}/正文.md`
5. **仿人工增强阶段**：润色完成后，主 session 统计正文虚词密度，与手工精修基准对比，对缺口类别用 SSoT 做定向补插后写回 `正文.md`
6. **交付阶段**：默认交付最终稿路径、各中间稿路径和 20 条来源清单，必要时再展示局部正文；交付后提示用户运行 `/novel-revise` 返修或 `/novel-sync` 同步
7. **保留阶段**：各风格中间稿默认保留，供回看、回滚和局部抽取

## 关键原则

1. **忠于大纲** — 在双层大纲约束内自由发挥台词和场景细节；若当前章节只有旧版 `第三人称精简剧情纲要`，也只在其范围内落文，不擅自改变剧情方向
2. **剧情点就是核对骨架** — 正文阶段必须固定细切为 20 个剧情点，并在所有稿件里显性保留
3. **风格分明** — 每种风格有明显差异，不是微调
4. **质量保证** — 每个版本和最终稿都经过校对和一致性检查
5. **自动合并有明确依据** — 最终稿按剧情点逐点择优，每个剧情点只选 1 个主来源版本，不做句群拼贴；合并规则支持章节级配置覆盖
6. **先看报告，再写正文** — `opus-report` 是放行条件和约束来源
7. **结构问题不上正文阶段硬修** — 需要新增/删除/重排关键剧情段时，回到 `novel-plan --revise`
8. **冗余描述在正文阶段清理** — 不把“水字数”主要寄希望于上游试写，正式成稿后必须再删一轮
9. **最终稿向项目正文气质收束** — 保留能提气的犀利观察、冷面反讽和精准吐槽，但不放任成段子腔或 POV 外调侃
10. **AI 味清理后再做润色** — AI 味检测由主 session 直接执行；检测完成后做高级感描写和武侠风动作润色，且不改剧情事实与因果
11. **润色后做仿人工增强** — 润色完成后，统计正文全部 7 类虚词密度并与手工精修基准对比，对低于基准的虚词类别用 SSoT 做定向补插（允许改写句式），使正文虚词分布接近手工精修水平
12. **返修与同步独立执行** — 正文交付后，用户运行 `/novel-revise` 进行返修，返修完成后运行 `/novel-sync` 进行同步；`novel-write` 不再包含返修和同步步骤
