# 输出规格

## 角色设定完整度验证

**用户审阅通过大纲后，自动扫描每个出场角色的设定完整度。**

大纲不需要包含角色的语言风格——台词设计是 novel-write 的职责。但需要确保每个涉及角色的设定文件存在且包含必填字段，否则 novel-write 无法正确创作。

**扫描规则**：

| 字段 | 标记 | protagonist | supporting/antagonist | 说明 |
|------|------|:-----------:|:--------------------:|------|
| 基本信息 | 🔴 | 必填 | 必填 | name, type, age, identity |
| 外貌描述 | 🔴 | 必填 | 必填 | 体型/面容/装束，至少一句话 |
| 核心性格 | 🔴 | 必填 | 必填 | 3-5 个核心性格特点 |
| 语言风格 | 🔴 | 必填 | 必填 | 说话方式/口头禅，novel-write 创作台词的直接依据 |
| 动机 | 🔴 | 必填 | 必填 | 角色的核心驱动力 |
| 修为境界 | 🔴 | 必填 | 必填 | 当前修为等级 |
| 人物关系 | 🔴 | 必填 | 必填 | 至少有一条关系记录 |
| 记忆点（视觉+行为+反差） | 🟡 | 必填 | 建议填写 | 角色辨识度的核心来源 |
| 语言记忆点 | 🟡 | 必填 | 建议填写 | 口头禅或标志性句式 |
| 在剧情中的作用 | 🟡 | 必填 | 可选 | 角色的叙事功能 |
| 行事风格 | 🟡 | 必填 | 可选 | 行为模式 |
| 梗/重复笑点 | 🟢 | 可选 | 可选 | 多角色互动才有 |
| 法宝/功法/战斗风格 | 🟢 | 可选 | 可选 | 战斗角色必填 |

**扫描执行方式**：

1. 从大纲「涉及角色」中提取所有角色名
2. 逐一读取 `memory/entities/characters/{角色名}.md`
3. 检查每个🔴字段的章节标题是否存在且内容非空
4. 输出完整度报告

**输出格式**：

```
📖 角色设定完整度报告

【苏寒染】主角 | 完整度：85%
  ✅ 基本信息 / 外貌描述 / 核心性格 / 语言风格 / 动机 / 修为境界 / 人物关系
  ⚠️ 记忆点（视觉）— 未填写视觉记忆点
  ⚠️ 记忆点（语言）— 未填写语言记忆点
  ❌ 无缺失必填字段

【凌绝尘】配角 | 完整度：60%
  ✅ 基本信息 / 核心性格 / 修为境界
  ⚠️ 外貌描述 — 内容为空
  ❌ 语言风格 — 缺失（novel-write 无法创作台词）
  ❌ 动机 — 缺失
  ❌ 人物关系 — 缺失

【临时NPC】无设定文件 → 通过动作展示，不安排重要台词
```

**处理策略**：

```
⚠️ 设定缺失预警

以下角色的必填字段缺失，会影响正文创作质量：

【凌绝尘】缺 3 个必填字段
  - 语言风格 ❌ → 无法创作差异化台词
  - 动机 ❌ → 无法判断角色行为合理性
  - 人物关系 ❌ → 无法确认与其他角色互动逻辑

建议操作：
   A. 现在补充（推荐）→ fork 出去用 /novel-discuss 补全
   B. 降级处理 → 本章该角色减少台词，用动作展示为主
   C. 暂时跳过 → 先保存大纲，写作前必须补全
```

**关键检查**：

- 如果某个角色的🔴字段全部缺失 → 强烈建议先做角色设计
- 如果只有语言风格缺失 → 本章该角色减少台词量，用动作展示
- 临时角色（不在知识图谱中的）可以简化处理：通过动作展示，不安排重要台词

## 上下文缓存生成

**在保存大纲的同时，生成章节级上下文缓存文件，供 novel-write 直接复用，避免重复收集。**

这一步将 context-collector 的工作前置到 plan 阶段，让 write 阶段可以跳过上下文收集，直接使用缓存。

**生成方式**：调用 context-collector agent（subagent_type: "context-collector"），prompt 如下：

```
为第 {chapter} 章生成上下文缓存。

大纲文件：chapters/ch-{chapter_padded}-outline.md

请根据大纲中的角色、地点、物品，收集完整的上下文信息。
将输出保存到 chapters/ch-{chapter_padded}-context.md。
```

**context-collector 会自动**：

1. 分析大纲中的涉及角色、地点、物品
2. 调用知识图谱搜索相关设定
3. 读取设定文件（仅相关段落，非全文）
4. 读取 past.md 最近剧情摘要
5. 读取 `memory/future/20-threads.md` 与相关卷卡 / arc 文件中的相关伏笔
6. 输出上下文缓存文件到 `chapters/ch-{chapter_padded}-context.md`

**上下文缓存文件格式**（`ch-xxxx-context.md`）：

```markdown
# 第 X 章上下文缓存

> 自动生成，novel-write 直接复用。设定变更后需重新生成。
> 生成时间：{timestamp}

## 出场角色

### 苏寒染（主角）
- 本章作用：{从大纲提取}
- 外貌：{从设定文件提取摘要，非全文}
- 语言风格：{从设定文件提取}
- 当前状态：{境界/位置/资源}
- 关键关系：{与本章其他角色的关系}

### 凌绝尘（配角）
- 本章作用：{从大纲提取}
- 外貌：{从设定文件提取摘要}
- 语言风格：{从设定文件提取}
- 记忆点：{视觉/行为/语言记忆点摘要}

## 涉及地点
### 地点名
- 关键描述：{从设定文件提取}
- 氛围/特点：{摘要}

## 相关世界观
- {修炼体系/势力/物品等，仅本章相关部分}

## 前情提要
- {最近 2-3 章的关键事件摘要}

## 伏笔状态
- 待回收：{本章需要回收的伏笔}
- 可埋设：{本章可以考虑埋设的伏笔}

## 写作注意事项
- {基于设定一致性需要注意的点}
- {基于大纲设计需要达成的目标}
```

**缓存失效规则**（novel-write 使用缓存前检查）：

- 角色设定文件修改时间 > 缓存生成时间 → 缓存失效，重新收集
- 知识图谱 `_graph.json` 修改时间 > 缓存生成时间 → 缓存失效
- worldbuilding.md 修改时间 > 缓存生成时间 → 缓存失效
- 当前章 `ch-xxxx-outline.md` 在缓存生成后被修订，且修订涉及出场角色 / 关键地点 / 伏笔状态 / 结束目标 → 缓存失效

**修订模式补充规则**：

- 如果只是措辞微调、标题微调、字数微调，不必重生成 context
- 如果剧情点顺序变了、角色出场变了、收尾点变了、伏笔处理变了，应视为 context 失效
- 修订完成后，优先先让用户确认修订版大纲，再尝试重生成 context
- 如果 context 重生成失败，要明确提示：旧 context 可能与新大纲不一致，`novel-write` 前必须补跑

## Opus 正文写作测试

**在用户确认大纲后，默认执行一次 Opus 正文写作测试。**

这一步不是点评大纲，而是让一个 Opus agent 根据当前大纲真实写一段正文，然后基于实际写作中的卡顿、脑补和节奏问题反推大纲缺陷。

**输出文件**：

- 测试正文：`chapters/ch-{chapter_padded}-opus-test.md`
- 反推报告：`chapters/ch-{chapter_padded}-opus-report.md`

**测试规则**：

1. 优先读取 `chapters/ch-{chapter_padded}-context.md`
2. 如果 context 缺失，允许降级为 outline-only 试写，但报告顶部必须写 `置信度：低`
3. 试写范围优先选取最容易暴露问题的连续 2-4 个剧情点
4. 测试正文必须是真正文，不少于 1000 字
5. 报告必须基于“写出来之后的卡点”反推问题，不能只做抽象评论

**说明**：

- 正式正文里的“冗余描述 / 水字数”清理由 `novel-write` 在成稿后单独执行
- `Opus` 测试报告可以顺手记录“信息过密”或“说明过多”这类节奏问题，但不再负责做最终删改判定

**测试正文格式**（`ch-xxxx-opus-test.md`）：

```markdown
# 第 X 章正文测试稿（Opus）

> 用于验证大纲可写性，不是最终成稿。
> 测试范围：剧情点3-5
> 生成时间：{timestamp}

【剧情点3：标题】

（正文）

【剧情点4：标题】

（正文）

【剧情点5：标题】

（正文）
```

**反推报告格式**（`ch-xxxx-opus-report.md`）：

```markdown
# 第 X 章大纲可写性报告（Opus）

> 基于正文试写反推问题，不替代人工判断。
> 置信度：高
> 测试范围：剧情点3-5
> 生成时间：{timestamp}

## 总体结论
- 判定：需小修后写
- 最大卡点：剧情点3进入剧情点4时，主角缺少做出决定的触发动作

## 写作中暴露的问题

### 1. 转场断裂
- 位置：剧情点3 → 剧情点4
- 表现：从“看到异常”直接跳到“决定跟进”，中间少一步动作或刺激
- 建议：补一个短桥接，让主角先做试探或收到反刺激

### 2. 角色声线发虚
- 位置：叶芷恒首次对话
- 表现：能写，但没有稳定的话语习惯，只能写成功能性对白
- 建议：在大纲或角色设定里补一句语言抓手

### 3. 信息投放过密
- 位置：剧情点4
- 表现：夜市规则、器坊体系、人物关系同时灌入，正文节奏发闷
- 建议：拆开，一部分后移到下一剧情点

## 被迫脑补的内容
- 主角为什么继续追查
- 周华顺为什么愿意立刻让步

## 回填建议
- 回填到大纲：主角转入主动的触发动作、周华顺让步的表面理由
- 回填到设定：叶芷恒的语言抓手

## 建议修订动作
- 保留：剧情点1-3的大方向
- 补桥：剧情点3后补 1 个选择动作
- 重排：把部分世界观说明后移
- 收尾：维持当前钩子，不必重写
```

**判定等级**：

- `可直接写`：Opus 能基本顺滑写出正文，结构和承接没有明显硬伤
- `需小修后写`：主线没问题，但桥接、信息分配、角色声线等局部问题需要补强
- `需先重做大纲`：Opus 试写时需要大量脑补，说明结构本身不可直接落文

**回流规则**：

- 如果判定为 `可直接写` → 允许进入 `/novel-write`
- 如果判定为 `需小修后写` → 优先进入 `novel-plan --revise`，做差分修订
- 如果判定为 `需先重做大纲` → 不建议继续正文创作，应先重建该段骨架

## 写作提示规范

`写作提示` 不是一个大段“写作指导”说明，而是给 `novel-write` 的**单行执行抓手**。

**定位**：

- 作用：告诉下游这段正文该怎么落笔、该把什么放大、什么不要解释
- 形式：每个剧情点默认附 1 行 `> ✏️ 写作提示：...`
- 长度：1-2 句，或 2-3 个短分句；默认不超过 2 行

**优先写什么**：

- 起手动作：这一段正文从哪个动作或镜头切入
- 镜头焦点：读者第一眼该注意哪个人、物或细节
- 放大项：哪种感官、关系张力或动作反差需要放大
- 对话质地：台词该偏冷、偏刺、偏松弛，还是要留停顿
- 禁解释项：哪些情绪、动机、关系不要直接说破

**不要写什么**：

- 不重复 `剧情概要`
- 不复述角色设定文件里的内容
- 不写“这一段要体现/展现/暗示/埋伏笔”
- 不写“注意控制节奏”“营造氛围”这类空话
- 不写成长段旁白式说明

**示例**：

- ✅ `> ✏️ 写作提示：先写吴仁停两秒再开口。消息别一股脑倒完。`
- ✅ `> ✏️ 写作提示：从顾青崖往碗里堆菜切入；先让动作带关系；别补心理。`
- ❌ `> ✏️ 写作提示：这一段要展现团队温暖，并为后续感情线埋伏笔。`
- ❌ `> ✏️ 写作提示：注意节奏和氛围，多一些细节描写。`

## 大纲保存格式

将完整的场景化剧情大纲保存到 `chapters/ch-xxxx-outline.md`，格式如下：

```markdown
# 第 X 章大纲：[标题]

## 基本信息
- 爽点类型：[类型] | 叙事模板：[模板]

## 起始状态
境界：... | 位置：... | 资源：...
关系：... | 关键变化：...

## 结束目标
1. [目标1]
2. [目标2]

## 涉及角色

### 苏寒染
- 本章作用：[作用描述]

### [其他角色]
- 本章作用：[作用描述]

## 剧情点

### 剧情点1：【标题】

地点：...
人物：...
氛围：...
情绪走向：...
场景类型：[类型] | 字数：[范围]
节奏职责：[可选]

剧情概要：[2-3句概要，海明威风格：短句、动作驱动、不解释动机；必须第三人称转述]

> ✏️ 写作提示：[1-2句或 2-3 个短分句；只给落文抓手，不重复剧情概要]

（...更多剧情点...）

## 伏笔计划
- 埋下：[伏笔] → 计划揭示：[位置]
- 揭示：[前置伏笔] → 本章揭示

```

## 修订模式输出

当 `novel-plan` 是基于已有 outline 做调整时，除了更新 `chapters/ch-xxxx-outline.md`，还要在对话中输出一份**修订摘要**，方便用户快速审阅本次改动。

**输出格式**：

```text
🧩 本次修订摘要

保留：
1. 剧情点1-2不变

改写：
1. 剧情点3增加试探和信息差

新增：
1. 新剧情点4，用于桥接叶芷恒出场

删除/弱化：
1. 删除原剧情点7中的解释性对话

重排：
1. 原剧情点5后移到剧情点6

联动影响：
1. 本章结尾改为"我要验验"
2. 下一章开头动作需要同步调整
```

**修订模式最少检查项**：

1. 起始状态是否仍与上一章结尾一致
2. 结束目标是否仍符合本章节奏职责
3. 伏笔计划是否需要延期/改回收位置
4. 涉及角色列表是否需要增删
5. 现有 `ch-xxxx-context.md` 是否失效
