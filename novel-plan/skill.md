# 剧情规划

## 目标

通过两种模式帮助用户完成单章工作：

- **新规划模式**：先定节奏职责、再定任务、最后展开
- **修订模式**：先读取已有大纲、再界定变更边界、最后做差分修改
- **正文测试**：用户确认大纲后，默认调用一个 Opus agent 真写一段正文，再根据试写结果反推大纲问题

### 新规划模式

#### 第一段：定节奏

1. 读取上下文并确认信息源优先级
2. 生成本章节奏定位（所属卷 / 承接 beat / 本章职责 / 本章不负责什么）

#### 第二段：定任务

3. 生成章节任务卡（本章主任务 / 必须推进的线 / 允许延期的线 / 建议收尾点）
4. 与用户确认本章范围和收尾点

#### 第三段：做展开

5. 一句话走向（10 选 1-2，或直接承接 future/ 已有规划）
6. 场景化方案（6 选 1，内部完成优化，不输出决策记录）
7. 剧情点大纲（写入文件，用户审阅）

### 修订模式

1. 读取当前章节已有大纲，提炼本章当前职责、起始状态、结束目标、剧情点骨架
2. 生成修订简报（保留什么 / 要解决什么 / 哪些改动会牵动前后章）
3. 给出 2-3 个修订策略，并推荐一个最稳的
4. 与用户确认变更边界（保留 / 删除 / 新增 / 重排 / 改收尾）
5. 生成修订清单，再更新已有大纲
6. 输出修订摘要和联动检查，必要时重生成 context

> **核心流程**：
> - 新规划模式：读取上下文 → 本章节奏定位 → 章节任务卡 → 确认收尾点 → 走向/方案展开 → 剧情点 → 写文件 → 用户确认
> - 修订模式：读取已有大纲 → 修订简报 → 修订策略 → 确认变更边界 → 修订清单 → 更新文件 → 联动检查 → 用户确认

## 使用方式

```bash
/novel-plan
/novel-plan --revise
```

## 执行流程

### 前置：初始化检查（仅首次）

1. 检查 `memory/future/00-index.md` 和 `memory/future/10-book.md` 是否存在。不存在则建议先 `/novel-bookplan`。
2. **确定章节编号**：从 `CLAUDE.md` 的「创作进度 → 当前章节」读取，或由用户指定（如 `/novel-plan 5`）。后续流程中章节文件一律使用四位补零格式：`ch-xxxx-outline.md` / `ch-xxxx-context.md`。
3. **确定信息源优先级**：
   - 新规划模式：
     - `memory/future/00-index.md` + `memory/future/10-book.md` + `memory/future/20-threads.md` + 相关 `30-volumes/` / `40-arcs/`：决定本章应推进的主线、所属卷、故事母型、事件边界、章节位置
     - `chapters/ch-{prev_padded}-outline.md`：决定上一章结束状态、章末钩子、待回收伏笔
     - `memory/past.md`：补充近期已发生剧情摘要，只作辅助，不覆盖前两者
     - `chapters/ch-{prev_padded}-context.md`：仅供 `novel-write` 使用，不作为本章规划主依据
   - 修订模式：
     - `chapters/ch-{current_padded}-outline.md`：第一信息源，决定当前章已经写好的骨架、起止状态、剧情点安排
     - 用户明确提出的修改目标：决定本次只改哪里，不改哪里
     - `memory/future/10-book.md` + `memory/future/20-threads.md` + 相关 `30-volumes/` / `40-arcs/`：检查修订后是否仍符合卷级/全书级职责
     - `chapters/ch-{prev_padded}-outline.md`：检查起始状态、章首承接是否被修订破坏
     - `memory/past.md`：仅补近期摘要
     - `chapters/ch-{current_padded}-context.md`：只用于判断缓存是否失效，不反向主导修订

如果高优先级文件缺失，明确提示缺哪一个；不要擅自用低优先级信息替代。

### 前置：模式判定

优先判断本次是**新规划模式**还是**修订模式**：

- 用户要从零设计下一章，或当前章还没有 outline → 新规划模式
- 用户明确说要在已有大纲上调整、修改、补桥接、改结尾、重排剧情点 → 修订模式
- 当前章已有 outline，且用户指出"这里太突兀/后半章不顺/收尾想改" → 修订模式

如果进入修订模式，默认**不**回到"10 个走向 + 6 个方案"的从零发散流程；只有在现有骨架已经失效，或用户明确要求大改时，才回退到新规划模式的展开阶段。

## 第一段：定节奏

### 第一步：读取上下文

输出当前进度、已完成章节、待处理伏笔、当前卷信息。

```text
📖 规划上下文

当前进度：
   卷：Vol-01（初入修真）
   已完成章节：第 10 章
   当前境界：炼气三层
   当前位置：凌霄宗外门

最近完成：
   第10章：演武场击败王师兄

待处理伏笔（来自 `memory/future/20-threads.md` 与 `memory/future/90-sync-tracker.md`，只读）：
   - 空灵根的秘密（Vol-03）
   - 神秘势力的关注（Ch 20）
```

同时整理起始状态（优先取上一章 outline 和 future/ 规划文件，`past.md` 只补摘要）：

```text
【起始状态】
   • 境界：炼气三层 | 位置：凌霄宗外门
   • 资源：灵石 50
   • 关系：敌对-王师兄，友好-李师姐
   • 近期：演武场切磋，展现不俗实力
```

**衔接检查**（自动执行，不需用户介入）：

读取上一章大纲 `chapters/ch-{prev_padded}-outline.md` 的结束目标、章末钩子、最后一个剧情点，与本章起始状态对比。发现不一致时在起始状态后标注：

```text
⚠️ 衔接提醒：
   上章结尾：苏寒染到达五丰镇城门（傍晚）
   本章起始：灵力载具停在城门外
   ✅ 一致
```

```text
⚠️ 衔接提醒：
   上章结尾：苏寒染在营地入睡
   本章起始：苏寒染在城门口排队
   ❌ 不一致——缺少出发/路上过渡，需补充
```

**章节伏笔衔接检查**（自动执行，不需用户介入）：

读取上一章大纲 `chapters/ch-{prev_padded}-outline.md` 的「伏笔计划」表格，筛选出状态为「⏳ 待回收」或「🔄 已移交追踪表」的伏笔。

如果存在待回收伏笔，输出：

```text
📦 上一章待回收伏笔

上一章（Ch{prev}）埋设了以下伏笔，本章是否需要回收：

1. [伏笔A] — 计划回收位置：[...]
2. [伏笔B] — 计划回收位置：[...]
...

请确认：
- 本章计划回收哪些伏笔？
- 未回收的伏笔是否继续延期？
```

如果 `memory/future/30-volumes/` 中已经存在对应卷卡，或 `memory/future/40-arcs/` 中已有当前 arc 文件，还要额外提取：

- 本章所属卷
- 该卷当前处于哪个 beat 区段
- 本章应该承担的节奏职责

如果这些信息缺失，明确提示：

```text
⚠️ 卷级节奏信息不足

我可以继续规划单章，但这会降低和整卷结构的贴合度。
更稳妥的做法：先回 /novel-bookplan 补该卷的节奏卡，再回来继续。
```

### 第二步：生成本章节奏定位

在章节任务卡之前，先输出本章在全书/分卷中的结构位置。

```text
📍 本章节奏定位

所属卷：Vol-02
主母型：Institutionalized
副母型：Fool Triumphant
卷内位置：中前段
承接的全书 beat：Fun and Games 区间
本章主职责：
1. 兑现这一卷的题材承诺
2. 推进主角在组织内部的站位变化
3. 为卷中点反转埋预热

本章不负责：
1. 提前揭开终极真相
2. 提前打完本卷 Boss
3. 一次性回收所有上一章伏笔
```

如果本章只是承上启下，也要直说，不要硬编成大转折章：

```text
本章类型：过桥章
结构价值：把角色从 A 状态送到 B 状态，同时保留钩子
```

## 第二段：定任务

### 第一步：生成章节任务卡

不要一上来让用户自己定义本章做什么。基于 future/ 的章节定位、分卷节奏卡、当前 arc、上一章结束状态、待回收伏笔，先提出一个可执行的章节任务卡，再让用户确认。

```text
📌 章节任务卡

本章主任务：进入五丰镇并完成第一次有效对接
必须推进的线：
1. 与周华顺建立联系
2. 展示夜市与鲁班坊体系
3. 让叶芷恒完成第一次有效出场

允许延期的线：
1. 空灵根暗线继续压后
2. 通讯阵更深层异常先不揭开

建议收尾点：苏寒染说"我要验验"
理由：
1. 完成被骗→反击的短弧线
2. 给下一章一个直接可写的开头动作
3. 不把夜市和周华顺线拖成半章尾巴
```

如果用户要改，优先改任务卡，不要直接跳到方案层。

### 第二步：确认本章范围和收尾点

在任务卡确认后，再确认本章收尾点和大致篇幅。不要让用户从零开始想收尾，而是给建议并说明为什么。

```text
📐 章节范围建议

建议收尾点：苏寒染说"我要验验"
字数预估：2800-3600 字（10 个左右剧情点）

这个收尾点可以吗？还是想提前/延后？
```

用户确认后，进入展开阶段。

## 修订模式（基于已有大纲调整）

### 第一步：读取当前章并生成修订简报

先读取当前章节的已有大纲，提炼：

- 本章当前职责
- 起始状态和结束目标
- 已经存在的剧情点骨架
- 已安排的伏笔回收/埋设
- 当前版本最可能出问题的点

输出格式：

```text
🩺 当前章修订简报

当前骨架：
1. 前半章先完成入场和信息对接
2. 中段发生被骗/试探
3. 结尾落在主角决定反击

建议保留：
1. 夜市初见的世界观展示
2. 周华顺的第一次有效接触

待解决问题：
1. 场景3到场景4转折过快
2. 叶芷恒出场偏功能化
3. 结尾钩子不够利落

可能受影响的联动：
1. 本章起始状态是否还承接上一章
2. 伏笔A是否仍然在本章回收
3. 下一章开头动作是否需要改
```

### 第二步：提出修订策略

不要让用户从零描述怎么改。先给 2-3 个修订策略，并推荐一个。

```text
🛠 修订策略

方案 A【微调版，最稳】
   保留 80% 现有剧情点，只补 1 个桥接场景，重写收尾点。

方案 B【中度重排，推荐】
   保留前半章入场和核心冲突，把中后段顺序重排，提前叶芷恒出场，让结尾更有钩子。

方案 C【后半章重做】
   前半章保留，后半章从被骗开始整体重写。
```

如果修订会明显牵动卷级节奏职责，要直接提醒：

```text
⚠️ 这已经不是单章微调，而是章节职责改动。

我可以继续帮你改这一章，但更稳妥的做法是先回 /novel-bookplan 看一下该卷节奏卡是否也要一起调整。
```

### 第三步：确认变更边界

确认时不要只问"想怎么改"。要把边界说具体：

```text
📏 本次修订边界

保留：
1. 开场到夜市入场
2. 周华顺第一次接触

删除/弱化：
1. 原场景4里的直接冲突

新增：
1. 一个被骗前的试探场景

重排：
1. 叶芷恒提前到中段出场

收尾：
1. 从"先记下了"改成"我要验验"

就按这个边界改，可以吗？
```

### 第四步：生成修订清单

用户确认后，再输出一份差分清单，再动文件：

```text
📋 修订清单

保留：
1. 剧情点1、2

改写：
1. 剧情点3 — 增加被骗前的试探和信息差

新增：
1. 新剧情点4 — 叶芷恒提前出场，先建立印象

重排：
1. 原剧情点5 后移到剧情点6

删除：
1. 原剧情点7 的重复解释段
```

### 第五步：更新已有大纲

更新规则：

1. 优先保留未受影响的段落，不为了"统一风格"整章重写
2. 重新编号被重排后的剧情点，避免旧编号残留
3. 如果结束目标、伏笔计划、涉及角色已被改动，要同步更新对应板块
4. 改的是局部，就只动局部；改的是中后段，不要顺手重写前半章

### 第六步：输出联动检查

修订完成后，必须追加一轮联动检查：

```text
🔎 修订后联动检查

上一章承接：
   ✅ 起始状态仍能接上

本章职责：
   ✅ 仍然承担 Fun and Games 的兑现

伏笔状态：
   ⚠️ 原计划本章回收的[伏笔A]已延期到下一章

下一章承接：
   ⚠️ 如果下一章仍按旧计划开头，需要把第一场改成"验货/对质"
```

如果已有 `chapters/ch-{current_padded}-context.md`，且本次修订影响了以下任一项，视为缓存可能失效：

- 出场角色变化
- 关键地点变化
- 伏笔状态变化
- 本章结束目标变化

此时在用户确认修订结果后，尝试重生成 context；失败则明确提示不要直接拿旧 context 写正文。

## 第三段：做展开（新规划模式）

### 第一步：一句话走向

以下流程默认用于**新规划模式**。修订模式只有在"现有骨架已失效"或"用户明确要求从头重做后半章/整章"时，才回到这一段。

先检查 `memory/future/40-arcs/` 中当前 arc 文件，或对应卷卡中，本章是否已有明确的事件规划或章节职责段：

**有明确规划** → 先复述职责，再确认是否沿用：

```text
future/ 当前 arc 中已有本章的明确规划：
   - 本章职责：Fun and Games 段的第一次兑现
   - 傍晚到达，对接周华顺
   - 夜市世界观展示
   - 被骗+对峙
   - 叶芷恒初遇

就按这个方向走？还是想调整？
```

**没有明确规划** → 走完整 10 个走向选择。基于当前剧情上下文（`past.md`、future/ 规划文件、上一章状态），给出 10 个可能的剧情走向，每个用一句话概括。直接询问，等待回答，不使用 AskUserQuestion。


**场景类型与字数**：

| 类型 | 字数 | 策略 |
|------|------|------|
| 日常 | 100-200 | 对话驱动，压缩写，留吐槽空间 |
| 信息揭示 | 200-400 | 信息藏在对话和动作里 |
| 情感 | 300-500 | 动作传情，放大写 |
| 战斗 | 300-600 | 密集短句，感官细节，放大写 |
| 转折 | 200-400 | 铺垫误导→一击翻盘 |
| 过渡 | 50-100 | 快速跳切，极简写 |
| 世界观展示 | 300-500 | 通过角色视线展开，每个细节有叙事功能（不能纯描写） |
| 初遇/重要出场 | 300-500 | 行为先于介绍，用互动定义角色，不说"来介绍一下" |

**核心原则**：高潮放大写，过渡快速过。总目标 2500-4000 字。

设计要点：

- 每个剧情点必须有情绪走向变化
- 剧情概要 2-3 句，足够让 `novel-write` 理解方向即可
- 剧情概要必须使用第三人称转述（使用角色名或“他/她/他们”），禁止第一人称（如“我/我们”）和第二人称（如“你/你们”）

- 写作提示只写 `novel-write` 立刻可执行的落文抓手，控制在 1-2 句或 2-3 个短分句
- 写作提示优先给：起手动作 / 镜头焦点 / 要放大的细节 / 对话质地 / 禁解释项
- 如果本章涉及上一章待回收伏笔，要在对应剧情点后标注处理方式：`已回收` / `延期` / `转移到后续章节`

**伏笔状态机规则**：

- `待埋设`：本章计划埋但尚未写进剧情点
- `待回收`：上一章或更早已埋，本章需要决定是否处理
- `已回收`：本章剧情点内已经明确安排回收动作
- `延期`：本章确认不处理，延到后续章节

`novel-plan` 只负责在本章大纲和本章规划说明里标记这些状态，不负责修改全局记忆；全局同步交给 `novel-sync`。

## 大纲写作规则

所有大纲文本必须遵守：

**剧情概要**：

- ✅ "他踢翻桌子。茶杯碎了。凌绝尘没有动。" — 具体动作，不说目的
- ❌ "他愤怒地踢翻桌子，展现出内心的不满" — 解释动机 + 叙事评论
- ✅ "残魂说阵法是垃圾。苏寒染没理他。继续拧螺丝。" — 用对话和动作推进
- ❌ "通过残魂的吐槽，侧面展现了修真界阵法的落后" — 旁白式总结
- ✅ "苏寒染推门进殿。她先看向香炉，再看向凌绝尘。" — 第三人称转述
- ❌ "我推门进殿，先看向香炉。" — 第一人称，不符合大纲约定
- 写 2-3 句具体动作/对话，不解释"这个场景的作用"

**氛围**：

- ✅ "灵石灯灭了两盏。走廊只剩最后一盏在闪。" — 具体细节暗示氛围
- ❌ "气氛紧张压抑，空气中弥漫着不安" — 空洞形容词堆砌
- 氛围用 2-4 个字足够，优先用具体物件/环境细节替代

**写作提示**：

- ✅ "残魂的吐槽要像老人抱怨手机难用" — 有画面感、有具体指向
- ✅ "先写吴仁停两秒再开口。消息不要一股脑倒完。" — 给出起手动作 + 节奏约束
- ✅ "从顾青崖往碗里堆菜切入。别补心理，先让动作把关系带出来。" — 给出镜头焦点 + 禁解释项
- ❌ "注意控制节奏，营造紧张氛围" — 废话
- ❌ "这一段要展现团队温暖，并为后续感情线埋伏笔" — 叙事功能评论，不是落文抓手
- 默认格式：`> ✏️ 写作提示：抓手A；抓手B；禁写项C`
- 只写 1-2 句，或 2-3 个短分句；不要写成长段说明
- 不重复剧情概要，不复述角色设定，不解释“这个场景的作用”

**禁止出现**：

- "展现出""体现了""暗示了""为…埋下伏笔" — 叙事功能评论（伏笔写在伏笔计划区）
- "仿佛""似乎""不禁""内心涌起" — 空洞心理描写
- 每个剧情点都有的工整标题 — 日常/过渡场景可省略
- 情绪走向写无实际变化的弧线 — 没变化就写"无变化"

## 关键原则

1. **先定节奏，再定任务，再做展开** — 本章先确认结构职责，再开始细化
2. **雪花法展开** — 一句话 → 场景化方案 → 剧情点大纲，逐层细化
3. **信息源有优先级** — future/ 的故事母型/分卷节奏卡 / arc 规划 和上一章 outline 优先于 `past.md`
4. **直接询问** — 不使用 AskUserQuestion，对话式沟通
5. **新规划模式文件只写一次** — 只有剧情点阶段写入大纲，用户可直接阅读和手动修改
6. **修订模式差分更新** — 需要调整时先确认边界，再改已有大纲，不默认整章重写
7. **方案先内化优化，再给用户看结果** — 不输出工具箱应用说明或决策记录
8. **聚焦单章** — 全书级与分卷级规划由 `novel-bookplan` 负责
9. **章节只承担职责，不重复造骨架** — 不把单章写成另一个全书规划
10. **派生文件不阻断主流程** — context 失败时保留大纲成果并明确提示补跑
11. **修订后必须做联动检查** — 至少检查前章承接、本章职责、伏笔状态、下一章起手
12. **先试写，再放行** — 大纲确认后默认跑一次 Opus 正文测试，用真实写作暴露问题

**禁止**：

- 使用 AskUserQuestion 工具
- 未经讨论直接修改文件
- 一次性抛出多个问题
- 只问问题不给方案
- 未经用户确认就保存大纲
- 用低优先级文件覆盖高优先级信息
- 跳过本章节奏定位和章节任务卡，直接进入散漫讨论
