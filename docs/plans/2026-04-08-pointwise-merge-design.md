# 逐个剧情点合并设计

**背景**

当前 `novel-write` 已经改成按风格逐个开启写作 Agent，避免 4 个风格版本在同一时刻一起发起请求。但最终稿合并阶段仍按“整章自动合并”的口径描述，容易让一次合并任务过大，也弱化了前面“内部写作单元”这层执行边界。

用户希望把最终稿合并也收束成更细粒度的过程：不是把整章一次性揉成一个版本，而是顺着正文阶段内部切分出来的剧情点 / 写作单元，逐个处理合并。

**目标**

在不改变最终交付形态的前提下，把合并流程明确为：

1. 最终稿仍然输出为连续章节正文，不暴露剧情点标题或内部写作单元标记。
2. 自动合并必须按内部写作单元顺序逐个进行，而不是整章一次性融合。
3. 每次只处理当前写作单元在 4 个风格版本中的对应内容，完成后再进入下一个写作单元。
4. 现有的风格吸收规则继续保留，只是从“整章层”下沉到“单元层”执行。

**设计决策**

### 1. 对外连续，对内逐段

这次不把内部写作单元重新暴露给用户，也不把最终稿改回带标题的剧情点拼接稿。写作单元只留在执行层，合并完成后只输出自然衔接后的连续章节正文。

### 2. 合并顺序跟随写作单元

自动合并从“整章选择主骨架并整体吸收”改成“按写作单元依次推进”。每一轮合并只处理当前写作单元对应的 4 个风格片段：

- 先确定当前单元的主骨架段落；
- 再从其余风格中吸收环境、动作张力、交锋感等局部表达；
- 合并完当前单元后，顺接到下一个单元。

这样可以把合并任务拆小，也更容易避免不同剧情段之间互相污染。

### 3. 风格规则继续沿用，但作用范围改成单元级

以下规则保持不变，只是从整章描述改成“当前写作单元”的局部决策：

- 卖报小郎君风负责主骨架和线索组织；
- 海明威风优先提供简洁环境句群；
- 大仲马风优先提供交锋与战斗推进；
- 火星引力风只做局部张力补强。

### 4. 文档与测试同步收口

这次仍然只改静态文档与测试，不新增脚本：

- `plugins/vibe-noveling/skills/novel-write/SKILL.md`
- `README.md`
- `tests/test_novel_write_workflow.py`

**验证**

新增静态文本测试，校验：

1. `novel-write` 明确写出“按写作单元逐个合并”。
2. `novel-write` 明确写出“每次只合并当前写作单元对应的 4 个风格片段”。
3. `novel-write` 明确说明最终稿仍为连续章节正文，不暴露内部切分。
4. README 对外说明同步为“逐个启动写作 Agent + 按写作单元逐个合并”。
