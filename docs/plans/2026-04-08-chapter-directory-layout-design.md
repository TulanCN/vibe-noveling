# 章节目录分卷重构设计

**背景**

当前工作流把所有章节相关产物都平铺在 `chapters/` 目录下，例如：

- `chapters/ch-0001-outline.md`
- `chapters/ch-0001-context.md`
- `chapters/ch-0001.md`
- `chapters/ch-0001-海明威.md`

这种结构在章节数量变多后会快速失去层次，也无法直接表达“这一章属于哪一卷”。更关键的是，这不是某个单独 skill 的偶然行为，而是整套提示词契约从 `novel-plan` 到 `context-collector`、`novel-write`、`novel-sync` 都把路径写死成了平铺文件名。

用户希望全面切换到分卷目录结构，不保留旧路径兼容。最终目标是让每一章都有自己的目录，并且目录上层显式带卷号。

**目标**

这次改动统一采用以下路径契约：

- `chapters/vol-01/ch-0001/大纲.md`
- `chapters/vol-01/ch-0001/上下文.md`
- `chapters/vol-01/ch-0001/Opus试写.md`
- `chapters/vol-01/ch-0001/Opus报告.md`
- `chapters/vol-01/ch-0001/缺失设定.md`
- `chapters/vol-01/ch-0001/正文.md`
- `chapters/vol-01/ch-0001/海明威.md`
- `chapters/vol-01/ch-0001/卖报小郎君.md`
- `chapters/vol-01/ch-0001/火星引力.md`
- `chapters/vol-01/ch-0001/大仲马.md`

同时满足以下约束：

1. `novel-plan`、`context-collector`、`novel-write`、`novel-sync` 全面切换到新路径。
2. 不保留旧平铺路径的读取或写入兼容逻辑。
3. 卷号是路径必填维度；如果当前章无法定位所属卷，要明确提示补卷信息，而不是继续沿用错误目录。
4. README 和测试要同步改成新结构，避免未来提示词回退。

**方案比较**

### 方案 A：卷目录 + 章节目录 + 短文件名

示例：

- `chapters/vol-01/ch-0001/大纲.md`
- `chapters/vol-01/ch-0001/正文.md`

优点：

- 目录语义最清晰，卷和章的归属一眼可见。
- 每章的全部产物自然归档到同一目录。
- 文件名短，阅读和引用成本低。

代价：

- 需要整套路径契约一起改，不是单文件修补。

### 方案 B：卷目录 + 章节目录，但保留旧长文件名

示例：

- `chapters/vol-01/ch-0001/ch-0001-outline.md`

优点：

- 迁移时与旧路径更相似。

代价：

- 目录名和文件名重复带章号，冗余明显。
- 既然已经全面切新结构，这种兼容式命名收益很低。

### 方案 C：只加卷目录，不加章节目录

示例：

- `chapters/vol-01/ch-0001-outline.md`

优点：

- 改动面略小。

代价：

- 每章产物仍然是平铺，只是少一层目录。
- 无法真正解决“每章相关文件散落”的问题。

**结论：采用方案 A。**

**设计决策**

### 1. 章节目录由卷号和章节号共同确定

每章的根路径统一表示为：

```text
chapters/vol-{volume_padded}/ch-{chapter_padded}/
```

后续所有读写都必须基于这个章节根路径展开，而不是再手写各类平铺文件名。

### 2. 中文短文件名作为正式契约

这次不继续沿用 `outline.md`、`context.md` 这种英文名，而是直接统一成中文短文件名。原因是这些文件本来就是面向中文创作工作流，中文文件名可读性更高，也更贴近用户在目录里直接浏览时的使用体验。

### 3. 卷定位失败时显式阻断

因为新结构把卷号放进了路径，卷信息缺失就不能再“先写到错误地方，之后再挪”。因此提示词需要明确：

- 优先从 `future/` 的卷级节奏卡或当前上下文判断所属卷；
- 如果无法确认，就提示用户先补卷信息或回到 `novel-bookplan`；
- 不允许回退写到 `chapters/ch-xxxx...` 旧路径。

### 4. 影响面是整条链路，不做局部补丁

这次不是只改 `novel-plan` 的输出路径。因为：

- `context-collector` 的输入输出都依赖大纲和上下文路径；
- `novel-write` 要读取大纲、上下文、Opus 报告，并写入最终稿和中间稿；
- `novel-sync` 要读取正文、大纲并更新同步状态；
- README 和项目结构说明也要同步反映新目录；
- 测试需要覆盖这条新路径契约，防止任何一个环节把旧路径写回来。

### 5. 静态测试守住路径契约

当前仓库主要通过静态文本测试守住提示词契约。这次继续沿用这个策略，新增断言确保：

1. `novel-plan` 使用 `chapters/vol-xx/ch-xxxx/大纲.md` 一类路径。
2. `novel-write` 的输入输出路径都切到新目录。
3. `context-collector` 与 `novel-sync` 不再提到旧平铺路径。
4. README 的项目结构与工作流说明同步改成分卷章节目录。

**验证**

至少执行两层验证：

1. 新增或更新的静态测试先失败后通过，证明路径契约确实被改动。
2. 跑完整个 `tests/` 目录，确认此前的工作流提示词约束没有被误伤。
