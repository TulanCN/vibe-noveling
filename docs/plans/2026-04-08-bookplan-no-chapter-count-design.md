# 全书规划去章节数绑定设计

**背景**

当前 `novel-bookplan` 的设计意图其实已经偏向“全书层看 beats、分卷层看职责、单章层再落地”，但提示词里仍残留了多处章节数量或章节范围的表达，例如“对应的大致章节范围”“开卷 3-5 章”“1-4 章 / 5-10 章”这类写法。这样会把全书规划错误地拉向“先定章数再排事件”，也会让 `future/` 在修订时被迫携带并不稳定的章位信息。

用户希望把契约进一步收口：全书大纲规划与 `future/` 事件管理阶段完全不考虑章节数量，只保留卷、beat、职责段、状态变化与事件依赖；章节编号只在 `novel-plan` 逐章规划时再引入。

**目标**

这次改动要明确以下边界：

1. `novel-bookplan` 不再要求或鼓励提供总章节数。
2. 全书 beat 蓝图不再记录“大致章节范围”，只记录卷位、关键事件、状态与是否锁定。
3. 分卷规划不再使用“3-5 章”“1-4 章”这类章节数表达，而改成职责段 / 前中后段 / 卷首卷中卷末等相对结构表达。
4. `future/` 维护阶段只管理结构信息和远期事件，不再把章节数量当作规划前提。
5. `novel-plan` 继续负责单章编号与单章落地，保留当前职责不变。

**设计决策**

### 1. 全书层保留体量感，但移除章数坐标

全书层仍然允许记录“预计字数或卷数”，因为这决定长篇体量和分卷密度；但不再出现总章节数、章节范围、某个 beat 大致落在第几章这类信息。全书蓝图只保留对 `novel-plan` 真正稳定有用的锚点：对应卷、关键事件、状态变化、锁定状态。

### 2. 分卷层从“章节段”改成“职责段”

卷内规划仍然要给 `novel-plan` 可执行的上游结构，但表达方式改为职责段，而不是章位段。比如“卷首建立新局面”“卷中兑现承诺感”“卷中后段反转预热”“卷末低谷与钩子”，避免把尚未落章的结构过早固化成数字。

### 3. reference 与主提示词同步收口

这次不能只改 `novel-bookplan/SKILL.md`，还要同步修改 `references/stc-hierarchy.md`，因为真正指导 agent 怎么切分结构的是 reference。如果只改主提示词，reference 里的“1-4 章 / 5-10 章”很容易让后续对话再次滑回章数模式。

### 4. 用静态测试守住新契约

当前测试还没有覆盖 `novel-bookplan`。这次新增静态断言，明确要求：

1. `novel-bookplan` 说明全书规划不预设章节数。
2. `novel-bookplan` 不再出现“大致章节范围”“大致章节”或“开卷 3-5 章”。
3. `stc-hierarchy` 使用职责段表述，而不是“1-4 章 / 5-10 章”这类示例。
4. README 对外文案同步说明 `novel-bookplan` 以卷和节拍为主，不预设章节数。

**验证**

执行静态文本回归测试，确认：

1. 新增的 `novel-bookplan` 契约断言先失败后通过。
2. 既有 `novel-write` 工作流测试继续通过，确保这次没有误伤单章与写作提示词。
