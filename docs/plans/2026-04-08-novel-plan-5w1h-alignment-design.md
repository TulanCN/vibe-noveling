# novel-plan 5W1H 对齐设计

**背景**

当前 `novel-plan` 已经把章节大纲收束成“第三人称精简剧情纲要”，这提升了人工审阅和手改体验。但在正式写大纲之前，流程只会做章节级任务确认和少量补问，没有强制把每个小剧情点的 `5W1H` 说清楚。结果是模型虽然能写出整体顺滑的大纲，局部细节却容易和用户真实预期产生细微偏差，尤其体现在人物为何这样做、转场发生在哪里、信息是在何时暴露、冲突如何具体落地这几类问题上。

用户希望保留“精简大纲”的外观，不把 `大纲.md` 再改回剧情点执行表；但在生成大纲前，必须先把每个小剧情点的 `5W1H` 和用户逐项对齐。

**目标**

在不破坏当前“精简大纲”契约的前提下，为 `novel-plan` 增加一个明确的前置对齐闸门：

1. 在生成正式大纲前，先拆出 3-8 个小剧情点。
2. 每个小剧情点都要明确 `Who / What / Why / Where / When / How`。
3. 这份 `5W1H` 对齐卡先在对话中给用户确认，而不是直接写入文件。
4. 用户确认后，才允许生成 `chapters/vol-{volume_padded}/ch-{chapter_padded}/大纲.md`。
5. 最终 `大纲.md` 继续保持“第三人称精简剧情纲要”，不回写 `5W1H` 表格，不重新膨胀成剧情点表。
6. 修订模式只重做受影响的小剧情点，不要求整章重新对齐。

**设计决策**

### 1. 新增“小剧情点对齐卡”作为强制中间层

这次不把 `5W1H` 融进最终大纲结构，而是在“场景化方案”与“设定完整度检查”之间插入一个固定步骤。

新的展开顺序应为：

1. 一句话走向
2. 场景化方案
3. 拆分 3-8 个小剧情点
4. 为每个小剧情点补齐 `5W1H`
5. 输出“小剧情点对齐卡”给用户确认
6. 确认后再进入设定完整度检查与正式大纲生成

这里的小剧情点只承担“澄清歧义”的职责，不等同于 `novel-write` 的写作单元，也不能回写到 `大纲.md`。

### 2. 对齐卡是对话产物，不是文件产物

用户确认前，`novel-plan` 只能在对话里展示如下结构：

```text
🧩 小剧情点对齐卡

1. [剧情点名称]
- Who：
- What：
- Why：
- Where：
- When：
- How：
```

这样做有两个原因：

- 保留现有大纲“适合人工直接阅读和手改”的优势；
- 避免重新引入“剧情点编号 + 执行字段”的旧问题。

因此，正式 `大纲.md` 中仍只保留“本章节奏定位 / 章节任务卡 / 起始状态 / 结束目标 / 第三人称精简剧情纲要 / 伏笔处理”等内容。

### 3. 对齐粒度固定为 3-8 个小剧情点

为了防止章节拆得过细，也避免过粗导致 `5W1H` 失去价值，这次将小剧情点数量约束为 3-8 个弹性范围。

- 少于 3 个：通常意味着颗粒度太粗，容易把多个关键变化揉在一起。
- 多于 8 个：通常意味着又在向“执行表”回摆，增加确认成本。

允许模型根据章节复杂度在这个范围内选择，但必须保证每个点都是真正需要用户确认的有效推进节点。

### 4. 用户确认是硬门槛

这次把“对齐卡确认”提升为强规则：

- 未确认 `5W1H` 对齐卡，不得写入 `大纲.md`
- 用户如果只改某几个点，只更新这些点后再继续
- 不允许跳过对齐卡直接进入正式大纲

这样能把偏差暴露在最早阶段，而不是等到大纲写完甚至试写时才返工。

### 5. 修订模式走局部对齐

修订模式不能因为要补一个桥接场景，就把整章重新拆解确认。因此规则应改为：

- 先识别本次修订影响了哪些小剧情点
- 只对新增、改写或受波及的点重做 `5W1H`
- 未受影响的点保持不动

这样既保留修订模式的“差分更新”原则，也把新的澄清能力接进来。

### 6. 测试只守契约，不模拟整段对话

当前仓库对这类 skill 工作流的保护主要依赖静态文本测试，因此本次继续沿用这一层：

- `tests/test_novel_write_workflow.py` 新增或扩展断言，校验 `novel-plan` 明确要求：
  - 先拆出 3-8 个小剧情点；
  - 每个点必须补齐 `Who / What / Why / Where / When / How`；
  - 先输出“小剧情点对齐卡”给用户确认；
  - 对齐卡不写入最终 `大纲.md`；
  - 修订模式只重做受影响的小剧情点。

这次不新增脚本，也不引入更复杂的自动化测试框架。

**影响范围**

- `plugins/vibe-noveling/skills/novel-plan/SKILL.md`
- `tests/test_novel_write_workflow.py`
- `README.md`

**验证**

完成后应满足：

1. `novel-plan` 文本里明确存在 `5W1H` 对齐步骤与“3-8 个小剧情点”的约束。
2. `novel-plan` 文本里明确写出“先确认对齐卡，再生成大纲”。
3. `novel-plan` 文本里明确写出“对齐卡不回写 `大纲.md`”。
4. `novel-plan` 文本里明确写出修订模式只重做受影响的小剧情点。
5. `README.md` 对外说明同步更新，避免用户误以为 `novel-plan` 会直接跳过剧情点澄清。
