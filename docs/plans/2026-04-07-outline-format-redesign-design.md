# 大纲格式重构设计

**背景**

当前工作流把“人类阅读的大纲”和“正文并行写作用的剧情点切分”混在同一份 `ch-xxxx-outline.md` 里，导致大纲层出现了过多只对写作 agent 有用的冗余信息，例如剧情点编号、情绪走向、场景类型、字数配额和写作提示。这种格式不利于作者手动修改。

**目标**

将章节大纲重构为一份更适合人工审阅和编辑的文档：

- 只保留本章节奏定位、任务卡、起始状态、结束目标、伏笔处理等高价值信息
- 主体改为第三人称精简剧情纲要
- 不再在 outline 中输出剧情点切分
- 将剧情点/写作单元的切分推迟到 `novel-write` 阶段内部完成
- 最终正文和中间稿默认输出为连续章节正文，不把内部切分暴露给用户

## 范围

本次只调整提示词与说明文档，不新增新的插件组件：

- `plugins/vibe-noveling/skills/novel-plan/SKILL.md`
- `plugins/vibe-noveling/skills/novel-plan/references/output.md`
- `plugins/vibe-noveling/skills/novel-write/SKILL.md`
- `README.md`
- `tests/test_novel_write_workflow.py`

## 设计决策

### 1. `novel-plan` 只产出人类可编辑的精简简纲

`ch-xxxx-outline.md` 改为以下结构：

- 章节基本信息
- 起始状态
- 结束目标
- 章节任务卡
- 第三人称精简剧情纲要
- 伏笔处理

其中“精简剧情纲要”使用 3-6 段短段落，按自然推进书写，但不做“剧情点1/2/3”式切分。

### 2. `novel-write` 在内部完成写作单元切分

正文阶段从精简简纲中抽取 6-12 个内部写作单元，仅供写作和并行分发使用：

- 不回写 `outline.md`
- 不要求用户审阅
- 不作为长期存档文件

这些写作单元只是一种执行细化，而不是新的用户面对格式。

### 3. 正文交付恢复为连续章节正文

无论是各风格中间稿，还是最终合并稿，都应输出为连续章节正文：

- 允许自然分段
- 必要时允许场景换行
- 不使用 `【剧情点N：名称】` 标题

内部写作单元用于执行，不暴露到最终稿。

### 4. `Opus` 试写也改按剧情段落而非剧情点取样

由于上游大纲不再显式给出剧情点，`Opus` 测试改为从精简剧情纲要中选取最容易暴露问题的连续剧情段落进行试写，并继续输出可写性报告。

### 5. 文档与测试同步

README 和测试必须同步表达以下新契约：

- `novel-plan` 生成精简剧情纲要
- `novel-write` 在正文阶段内部切分
- 最终正文不保留剧情点标题

## 风险与约束

1. `novel-plan` 与 `novel-write` 的输入契约必须同时修改，否则会出现“上游不再给剧情点、下游仍要求剧情点”的断裂。
2. 由于这是提示词工作流，回归保护主要依赖文本测试与 grep 校验，而不是运行时单元测试。
3. 现有仓库里已经有 `novel-write` 自动合并相关未提交改动，因此本次必须只做增量修改，避免覆盖用户已有变化。

## 验证策略

1. 文本测试校验 `novel-plan` 明确提到“第三人称精简剧情纲要”和“不再输出剧情点切分”。
2. 文本测试校验 `novel-write` 明确提到“正文阶段内部切分写作单元”与“最终稿输出为连续章节正文”。
3. README 同步更新为“精简剧情纲要 + 正文阶段内部切分”的公开说明。
4. 额外用 `rg` 检查旧的强制剧情点输出要求是否仍残留在关键用户路径中。
